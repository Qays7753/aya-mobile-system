## 📐 Device-Agnostic UX Contract

> هذا العقد إلزامي: كل واجهة يجب أن تعمل على الهاتف + التابلت + اللابتوب + الديسكتوب بدون استثناء.

### قواعد تخطيط حسب الجهاز

| الجهاز | التخطيط | التنقل | المدخلات |
|--------|---------|--------|----------|
| هاتف (`360px+`) | عمود واحد | قائمة جانبية عبر زر ☰ | Touch-first |
| تابلت (`768px+`) | عمودان عند الحاجة | قائمة قابلة للطي | Touch + Keyboard |
| لابتوب (`1024px+`) | تخطيط كامل متعدد الأعمدة | قائمة جانبية ثابتة | Keyboard-first + Mouse |
| ديسكتوب (`1280px+`) | تخطيط تقارير كامل متعدد اللوحات | قائمة جانبية ثابتة | Keyboard-first + Mouse |

### معايير قابلية الاستخدام

1. الحد الأدنى لأبعاد الأزرار التفاعلية على الهاتف: `44x44px`.
2. المسافة بين عناصر اللمس: `8px` كحد أدنى.
3. لا يسمح بـ Horizontal Scroll لأي شاشة تشغيلية عند `360px`.
4. النماذج الطويلة على الهاتف تُقسم إلى أقسام واضحة بدل صفحة مزدحمة واحدة.
5. جداول البيانات على الهاتف تتحول إلى بطاقات (Cards) عند الحاجة، وليس جدولًا مضغوطًا غير قابل للقراءة.
6. شاشة POS يجب أن تدعم:
   - Touch mode (الهاتف/التابلت)
   - Keyboard mode (اللابتوب)
7. تغير الاتجاه (Portrait/Landscape) لا يكسر الشاشة ولا يخفي أزرار الإجراء الرئيسية.

### قابلية التثبيت (Installability UX)

1. إظهار خيار "تثبيت التطبيق" بشكل غير مزعج للمستخدم عند توفره.
2. تجربة التطبيق المثبت يجب أن تحافظ على نفس الصلاحيات والسلوك الأمني.
3. لا يتم تفعيل أي سلوك Offline مالي بعد التثبيت.

## 🎯 مبادئ التصميم للمستخدم البسيط

> **القاعدة الذهبية:** إذا احتاج الموظف أن يقرأ دليلاً ليفهم الشاشة — التصميم فاشل.

| المبدأ | التطبيق |
|--------|---------|
| **أزرار كبيرة** | 48px حد أدنى، 56px في POS |
| **نص واضح** | الخط 16px كحد أدنى، المبالغ 20px+ **بخط عريض** |
| **أيقونات + نص دائماً** | لا أيقونات وحدها (المستخدم قد لا يفهمها) |
| **ألوان واضحة المعنى** | 🟢 Emerald = نجاح/تأكيد، 🟡 Amber = تحذير، 🔴 Red = خطأ/حذف |
| **تأكيد قبل العمليات الحساسة** | حوار تأكيد: المرتجع، الإلغاء، الحذف |
| **لا صفحات فارغة** | دليل إرشادي بدلاً من "لا بيانات" |
| **كل شيء قابل للنقر/اللمس** | لا اعتماد على لوحة المفاتيح |
| **عمليات يومية ≤ 5 خطوات** | البيع النقدي: بحث → إضافة → دفع → تأكيد → ملخص |
| **صوت إتمام البيع** | beep قصير عند نجاح الفاتورة (ADR-025) |
| **Loading واضح** | spinner مع نص "جاري التحميل..." |
| **وضوح التزامن** | عند تعارض مخزون بسبب بيع متزامن: رسالة فورية مع اقتراح إعادة المحاولة |

---

## 🛒 تجربة مستخدم نقطة البيع (POS UX) - قرارات معتمدة

بناءً على التوجيهات الإدارية، شاشة نقطة البيع (POS) يجب أن تلتزم دائماً بالقواعد الأربع الجوهرية التالية:

1. **شاشة "عمياء" (Blind POS):** ممنوع منعاً باتاً عرض إجمالي مبيعات اليوم، أو تكلفة الشراء، أو الأرباح في شاشة موظف الـ POS. الشاشة مخصصة للبيع فقط (للحفاظ على سرية الأداء المالي).
2. **نافذة الدفع الموحدة (Unified Mixed Payments Popup):** لا يوجد تعدد للشاشات عند الدفع. عند الضغط على زر "دفع"، تظهر نافذة منبثقة (Popup) واحدة تحتوي على حقول لكل طرق الدفع (كاش، فيزا، دين) وتُمكن الموظف من إدخال الدفع المختلط مباشرة مع حساب المتبقي فورياً.
3. **التصنيفات السريعة (Category Chips):** يجب توفر أزرار علوية سريعة (أجهزة، إكسسوارات، شرائح، خدمات صيانة) لفلترة شبكة المنتجات بنقرة واحدة بدون الحاجة للبحث النصي دائماً.
4. **شبكة الأكثر مبيعاً (Quick Add Grid):** تخصيص مساحة بارزة في الشاشة تعرض أكثر المنتجات مبيعاً لتتم إضافتها للفاتورة بنقرة واحدة لتسريع عملية البيع بشكل جذري.

> **تنفيذ أمني إلزامي:** قاعدة Blind POS لا تعتمد على UI فقط، بل تُفرض أيضاً عبر طبقة البيانات (Views آمنة + RLS) لمنع قراءة الحقول الحساسة مباشرة.

---

## 🗣️ لغة الواجهة للمستخدم البسيط

- استخدم جملة قصيرة مباشرة (فعل + نتيجة): "تم حفظ الفاتورة بنجاح".
- تجنب المصطلحات التقنية في الواجهة (مثل: transaction, idempotency).
- عند الخطأ: اشرح المشكلة + الخطوة التالية ("تأكد من الشبكة ثم أعد المحاولة").
- لا تعرض أكثر من إجراء رئيسي واحد في كل شاشة حرجة (POS/تسديد/مرتجع).

## ❌ كتالوج رسائل الخطأ

| الكود | الرسالة (عربي) | متى تظهر |
|-------|----------------|----------|
| `ERR_STOCK_INSUFFICIENT` | المخزون غير كافٍ — {اسم المنتج}: المتوفر {عدد} فقط | بيع بكمية أكبر من المتوفر |
| `ERR_DISCOUNT_EXCEEDED` | الخصم يتجاوز الحد المسموح (10%) | موظف POS يخصم أكثر من 10% — **يمنع العملية (ROLLBACK)** |
| `ERR_DEBT_LIMIT_WARNING` | ⚠️ تنبيه: تجاوز حد الدين — الرصيد الجديد: {مبلغ} د.أ | بيع بالدين يتجاوز الحد — **إشعار فقط، لا يمنع العملية** (ADR-006) |
| `ERR_CANCEL_HAS_RETURN` | لا يمكن إلغاء فاتورة عليها مرتجع — ألغِ المرتجع أولاً | إلغاء فاتورة مرتبطة بمرتجع |
| `ERR_CANCEL_ALREADY` | هذه الفاتورة ملغاة مسبقاً | محاولة إلغاء فاتورة ملغاة |
| `ERR_CANCEL_REASON` | يجب إدخال سبب الإلغاء (5 أحرف على الأقل) | إلغاء بدون سبب |
| `ERR_NETWORK` | لا يوجد اتصال بالإنترنت — تأكد من الشبكة وأعد المحاولة | انقطاع الاتصال |
| `ERR_PAYMENT_MISMATCH` | مجموع طرق الدفع ({مبلغ}) لا يساوي الإجمالي ({مبلغ}) | دفع مختلط غير متوازن |
| `ERR_RETURN_QUANTITY` | كمية المرتجع ({عدد}) أكبر من الكمية المباعة ({عدد}) | إرجاع أكبر |
| `ERR_DEBT_OVERPAY` | مبلغ التسديد أكبر من الرصيد المتبقي ({مبلغ} د.أ) | تسديد أكثر من الدين |
| `ERR_UNAUTHORIZED` | لا تملك صلاحية لهذه العملية | موظف يحاول عملية Admin |
| `ERR_PRODUCT_EXISTS` | يوجد منتج بنفس الاسم بالفعل | إضافة منتج مكرر |
| `ERR_IDEMPOTENCY` | هذه العملية مسجلة مسبقاً — لا حاجة لتكرارها | ضغط إتمام مرتين |

| `ERR_CONCURRENT_STOCK_UPDATE` | تغير المخزون أثناء تنفيذ العملية — راجع الكمية وأعد المحاولة | بيع متزامن على نفس المنتج |
| `ERR_SERVER` | حدث خطأ — أعد المحاولة أو تواصل مع أحمد | خطأ غير متوقع |

## 🌐 سياسة بدون Offline

- النظام يعمل Online فقط.
- عند انقطاع الشبكة: تظهر رسالة `ERR_NETWORK` وتُوقف العملية.
- لا يوجد حفظ محلي مؤقت للعمليات المالية.

## 🔄 استرداد الأخطاء والمحاولة مرة أخرى (Error Recovery)

| السيناريو | ماذا يحدث | ماذا يرى المستخدم |
|----------|-----------|-------------------|
| **فشل العملية بالكامل** (DB ROLLBACK) | العملية تلغى تلقائياً — لا شيء يُحفظ | رسالة `ERR_SERVER` + زر "أعد المحاولة" |
| **انقطاع الاتصال أثناء العملية** | بفضل `idempotency_key`: إعادة المحاولة مع نفس المفتاح تُرجع النتيجة السابقة إذا نجحت، أو تعيد التنفيذ إذا لم تكتمل | رسالة `ERR_NETWORK` + زر "أعد المحاولة" (يرسل نفس `idempotency_key`) |
| **تعارض مخزون متزامن** | DB ROLLBACK تلقائي | رسالة `ERR_CONCURRENT_STOCK_UPDATE` + زر "أعد المحاولة" (مفتاح جديد) |
| **خطأ غير متوقع مستمر** | DB ROLLBACK + لا تأثير على البيانات | رسالة `ERR_SERVER` + "تواصل مع أحمد" |

**القاعدة العامة:** جميع العمليات المالية تعمل داخل PostgreSQL Transaction — إما تنجح بالكامل أو تفشل بالكامل. لا يوجد "فشل جزئي" يُفسد البيانات.

## ⚠️ سلوك تعديل الفاتورة (ADR-026)

**القرار:** يمكن تعديل الفاتورة بعد إتمامها، بشروط:
- يجب إدخال **سبب التعديل** (إلزامي)
- يُسجل في **Audit Log** (القيم قبل/بعد)
- يُنشئ **إشعار** لأحمد
- يُعدّل **Ledger Entries** تلقائياً (عكس القديم + إنشاء جديد)
- يُعدّل **المخزون** إذا تغيرت الكميات
- لا يمكن تعديل فاتورة **ملغاة** أو عليها **مرتجع**
