## 🔴 مخاطر تقنية

| # | المخاطر | الخطورة | المعالجة |
|---|---------|---------|----------|
| RISK-01 | **عدم وجود آلية إعادة حساب الأرصدة** — إذا حصل خلل في `accounts.current_balance` | 🔴 | إنشاء دالة `recalculate_account_balance()` تُقارن الرصيد المُخزّن مع المحسوب من `ledger_entries` |
| RISK-02 | **`debt_customers.current_balance`** يمكن أن يصبح سالباً | 🟡 | مقصود للسماح بتصحيح أخطاء — لكن يجب إضافة تنبيه عند القيم السالبة |
| RISK-03 | **Rate Limiting** مذكور في `16_Error_Codes.md` لكن آلية التنفيذ غير موثقة | 🟡 | توثيق الآلية في `13_Tech_Config.md`: حد 100 طلب/دقيقة لكل مستخدم عبر Vercel Edge Middleware |
| RISK-04 | **التزامن في POS** — بيع نفس المنتج من جهازين | 🔴 | مُعالج عبر `SELECT FOR UPDATE` (ADR-023) — يجب اختبار سيناريو UAT-03 بدقة |
| RISK-05 | **الاعتماد الكامل على الإنترنت** — لا يوجد Offline Mode | 🟡 | قرار تصميمي (ADR-021). المعالجة: `idempotency_key` + Toast واضح + زر "أعد المحاولة" |
| RISK-14 | **جهاز مفقود/مسروق فيه جلسة نشطة** — المهاجم يمكنه استخدام النظام | 🔴 | إنهاء جميع جلسات المستخدم عبر Supabase Auth Admin API + تغيير كلمة المرور فوراً (SOP-34) |

## 🧭 مخاطر إضافية مكتشفة في Final Go/No-Go (محدّثة)

| # | الخطر | الحالة | المعالجة الموثقة |
|---|-------|--------|------------------|
| RISK-09 | تسرب بيانات حساسة للـ POS عبر قراءة مباشرة للجداول | ✅ مغلق توثيقياً | فرض Blind POS عبر Views آمنة + منع القراءة المباشرة للأعمدة الحساسة |
| RISK-10 | Deadlock محتمل في البيع المتزامن متعدد العناصر | ✅ مغلق توثيقياً | Lock Ordering (`product_id ASC`) + Retry Policy + UAT-21b |
| RISK-11 | تعارض عقود الحقول بين الملفات المرجعية | ✅ مغلق توثيقياً | توحيد العقود (`refund_account_id`, `payment_id`, `is_paid/unit_cost`) |
| RISK-12 | غموض محاسبي في مرتجع فواتير الدين بعد سداد جزئي | ✅ مغلق توثيقياً | قاعدة توزيع آلية + كود خطأ `ERR_RETURN_REFUND_ACCOUNT_REQUIRED` |
| RISK-13 | عدم اكتمال ربط عمليات DB بمسارات API | ✅ مغلق توثيقياً | مصفوفة رسمية DB ↔ RPC ↔ API في ملف `13` |
