# آية موبايل - سيناريوهات Dry Run المالية
## 26) Financial Dry Run Scenarios (Pre-Build)

---

## الهدف

تنفيذ تمرين محاكاة مالي قبل كتابة الكود للتأكد من أن العقود المالية مكتملة ومغلقة محاسبيًا.

---

## نطاق Dry Run

يغطي 5 سيناريوهات حرجة:
1. بيع مختلط (نقدي + فيزا)
2. بيع بالدين
3. مرتجع جزئي
4. تسديد دين FIFO
5. إلغاء فاتورة

---

## متطلبات مسبقة

1. وجود منتج واحد على الأقل بمخزون كافٍ وسعر بيع وتكلفة.
2. وجود حسابات: `cash`, `visa`.
3. وجود عميل دين واحد (`debt_customer`).
4. وجود مستخدم Admin ومستخدم POS.
5. تفعيل سياسات ADR-042/043/044 في التوثيق.

---

## DR-01: بيع مختلط

**المدخلات**
- سلة: منتج/منتجان.
- دفعات: جزء نقدي + جزء فيزا.
- `idempotency_key` فريد.

**التوقعات**
1. إنشاء: `invoices`, `invoice_items`, `payments`.
2. تحديث: `products.stock_quantity`.
3. إنشاء قيود `ledger_entries` لكل حساب دفع.
4. نجاح معادلة التوازن:
`SUM(payments.amount) + debt_amount = total_amount`.

**فشل متوقع**
- عدم تطابق المدفوعات → `ERR_PAYMENT_MISMATCH`.

---

## DR-02: بيع بالدين

**المدخلات**
- سلة بيع.
- `debt_amount > 0`.
- `customer_id` موجود.

**التوقعات**
1. إنشاء `debt_entries` بقيمة الدين.
2. تحديث `debt_customers.current_balance += debt_amount`.
3. قيود `ledger_entries` فقط للمبلغ المدفوع فعليًا (إن وجد).
4. منع تسعير العميل: السعر يُسحب من DB فقط.

**فشل متوقع**
- تجاوز سياسة/صلاحية → `ERR_UNAUTHORIZED` أو `ERR_API_ROLE_FORBIDDEN`.

---

## DR-03: مرتجع جزئي

**المدخلات**
- فاتورة فعالة تحتوي عناصر غير مرجعة بالكامل.
- إرجاع عنصر/كمية جزئية.

**التوقعات**
1. إنشاء `returns`, `return_items`.
2. تحديث `invoice_items.returned_quantity`.
3. تحديث `invoices.status` إلى `partially_returned` عند اللزوم.
4. إعادة مخزون العنصر المرجع.
5. إنشاء قيد `ledger_entries` صادر لحساب الإرجاع.

**حالة الدين الخاصة**
- المرتجع يخصم من الدين غير المسدد أولًا.
- إذا وُجد `cash_refund > 0` يصبح `refund_account_id` إلزاميًا.

**فشل متوقع**
- كمية إرجاع أعلى من المسموح → `ERR_RETURN_QUANTITY`.

---

## DR-04: تسديد دين FIFO

**المدخلات**
- عميل عليه أكثر من `debt_entry` غير مسدد.
- دفعة جزئية واحدة.

**التوقعات**
1. إنشاء `debt_payments`.
2. توزيع التسديد FIFO على `debt_entries` (الأقدم أولًا).
3. إنشاء `debt_payment_allocations` لكل توزيع.
4. تحديث `debt_customers.current_balance` بالقيمة المسددة.
5. إنشاء قيد `ledger_entries` وارد للحساب المستلم.

**فشل متوقع**
- محاولة تسديد أعلى من الرصيد → `ERR_DEBT_OVERPAY`.

---

## DR-05: إلغاء فاتورة

**المدخلات**
- فاتورة غير ملغاة، بدون مرتجعات مرتبطة.
- `cancel_reason` إلزامي.

**التوقعات**
1. تحديث `invoices.status = cancelled`.
2. عكس حركات المخزون.
3. إنشاء قيود عكسية في `ledger_entries`.
4. عكس أثر الدين (إن وجد) على `debt_entries` و`debt_customers`.
5. تسجيل العملية في `audit_logs`.

**فشل متوقع**
- وجود مرتجع مرتبط بالفاتورة → `ERR_CANCEL_HAS_RETURN`.

---

## نموذج توثيق نتيجة كل Dry Run

```md
### DR-XX Result
- المنفذ:
- التاريخ:
- الحالة: Pass / Fail
- أدلة:
  - invoice_number / return_number / receipt_number
  - affected tables
  - error code (if fail)
- ملاحظات:
- قرار: Continue / Block
```

---

## معايير النجاح النهائية

1. جميع السيناريوهات الخمسة `Pass`.
2. لا يوجد تضارب في معادلات التوازن المالي.
3. جميع حالات الفشل ترجع `ERR_*` مطابق للكتالوج.
4. لا توجد كتابة مباشرة من المتصفح خارج API.
