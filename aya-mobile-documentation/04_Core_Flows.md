## 🔄 تدفق 1: البيع النقدي السريع

```
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│  البدء  │────▶│  البحث  │────▶│ إضافة   │────▶│  الدفع  │────▶│  إتمام  │
│         │     │ عن منتج │     │ للفاتورة│     │  نقدي   │     │  البيع  │
└─────────┘     └─────────┘     └─────────┘     └─────────┘     └─────────┘
     │               │               │               │               │
     ▼               ▼               ▼               ▼               ▼
  فتح POS        كتابة اسم      اختيار كمية    إدخال المبلغ    تسجيل الفاتورة
  أو اختصار      أو اختيار       (افتراضي 1)    تأكيد           تحديث مخزون
  / للبحث        من القائمة                       تسجيل حركة    تسجيل Audit
                                                                   Log
```

**الخطوات التفصيلية:**

| الخطوة | الإجراء | النظام يفعل |
|--------|---------|-------------|
| 1 | المستخدم يفتح POS | يعرض شاشة فارغة |
| 2 | يبحث عن المنتج | يظهر نتائج البحث |
| 3 | يختار المنتج | يضاف للفاتورة (السعر يُعرض من products.sale_price — للعرض فقط) |
| 4 | يكرر 2-3 لكل منتج | - |
| 5 | يختار "نقدي" | يفتح حقل المبلغ |
| 6 | يدخل المبلغ | يحسب الباقي |
| 7 | يضغط "إتمام" | يُرسل product_id + quantity فقط إلى API Route (ADR-043) |
| 8 | - | API Route يستدعي create_sale() بـ service_role → يحسب الأسعار من DB → ينشئ فاتورة + يحدث مخزون + يسجل حركة |
| 9 | - | يعرض ملخص الفاتورة ويُسجل العملية |

```
└─────────┘     └─────────┘     └─────────┘     └────┬────┘
                                                     │
                              ┌──────────────────────┼──────────────────────┐
                              │                      │                      │

                         │ 1000    │           │ 2000    │           │ 1000    │
                         └─────────┘           └────┬────┘           └─────────┘
                                                     │
                                                     ▼
                                              ┌─────────┐
                                              │ حساب    │
                                              │ العمولة │
                                              │ (2%=40) │
                                              └────┬────┘
                                                   │
                              ┌────────────────────┼────────────────────┐
                              │                    │                    │

```

**الخطوات التفصيلية:**

| الخطوة | الإجراء | النظام يفعل |
|--------|---------|-------------|

| 2 | اختيار "دفع مختلط" | يفتح شاشة التقسيم |
| 3 | إدخال مبلغ نقدي | يحسب المتبقي |
| 4 | إدخال مبلغ فيزا | يحسب المتبقي + العمولة |
| 5 | إدخال مبلغ محفظة | يتحقق من التوازن |
| 6 | تأكيد | ينشئ 3+ Ledger Entries |
| 7 | - | كل Entry مرتبط بالفاتورة |

**الخطوات التفصيلية:**

| الخطوة | الإجراء | النظام يفعل |

## 🗂️ OP-01: ربط عملية البيع بالجداول (Sale Flow → Tables)

### الجداول المتأثرة

| الترتيب | الجدول | العملية | الشرط |
|---------|--------|---------|-------|
| 1 | `products` | `SELECT FOR UPDATE` | قفل المنتجات لمنع بيع متزامن |
| 2 | `invoices` | `INSERT` | دائماً |
| 3 | `invoice_items` | `INSERT` (عدة سجلات) | سجل لكل منتج |
| 4 | `payments` | `INSERT` (عدة سجلات) | سجل لكل طريقة دفع |
| 5 | `products` | `UPDATE stock_quantity` | خصم الكمية المباعة |
| 6 | `ledger_entries` | `INSERT` (عدة سجلات) | قيد لكل طريقة دفع |
| 7 | `debt_entries` | `INSERT` | فقط إذا debt_amount > 0 |
| 8 | `debt_customers` | `UPDATE current_balance` | فقط إذا debt_amount > 0 |
| 9 | `notifications` | `INSERT` | إذا تجاوز حد الدين أو الخصم |

### التنفيذ خطوة بخطوة

#### الخطوة 1: التحقق (Validation)
```sql
-- قفل المنتجات لمنع بيع متزامن (PESSIMISTIC LOCK)
SELECT id, stock_quantity, sale_price, cost_price
FROM products
WHERE id IN (items[].product_id)
ORDER BY id ASC
FOR UPDATE;

-- التحقق 1: المخزون
IF stock_quantity < requested_quantity → ERR_STOCK_INSUFFICIENT + ROLLBACK

-- التحقق 2: الخصم (POS فقط)
IF discount_percentage > 10% AND role = 'pos' → ERR_DISCOUNT_EXCEEDED + ROLLBACK

-- التحقق 3: idempotency
IF idempotency_key EXISTS → ERR_IDEMPOTENCY + ROLLBACK

-- التحقق 4: مجموع الدفعات
IF SUM(payments[].amount) + debt_amount ≠ total_amount → ERR_PAYMENT_MISMATCH + ROLLBACK

-- التحقق 5: invoice_date = CURRENT_DATE تلقائياً (ADR-034 مُعدّل: لا Backdating مطلقاً)
-- لا حاجة للتحقق — الحقل يُعبأ سيرفرياً بـ CURRENT_DATE ولا يُقبل من العميل
```

#### الخطوة 2: إنشاء الفاتورة
```sql
INSERT INTO invoices (
  invoice_number,    -- AYA-2026-00001 (auto-generated)
  invoice_date,      -- CURRENT_DATE دائماً (ADR-034: لا Backdating)
  customer_name,     -- اختياري
  customer_phone,    -- اختياري
  subtotal,          -- SUM(items.quantity × items.unit_price)
  discount_amount,   -- SUM(items.discount_amount)
  total_amount,      -- subtotal - discount_amount
  debt_amount,       -- المبلغ المؤجل (0 إذا لا دين)
  debt_customer_id,  -- NULL إذا لا دين
  status,            -- 'active'
  created_by,        -- UUID المستخدم الحالي
  pos_terminal_code, -- كود الجهاز (اختياري)
  idempotency_key    -- UUID فريد لمنع التكرار
) RETURNING id, invoice_number;
```

#### الخطوة 3: إنشاء عناصر الفاتورة (لكل منتج)
```sql
INSERT INTO invoice_items (
  invoice_id,          -- من الخطوة 2
  product_id,          -- UUID المنتج
  quantity,            -- الكمية المطلوبة
  unit_price,          -- سعر البيع الحالي
  cost_price_at_time,  -- ← products.cost_price (لقطة لحساب الربح لاحقاً)
  discount_percentage, -- نسبة الخصم (0-100)
  discount_amount,     -- quantity × unit_price × (discount_percentage / 100)
  total_price          -- (quantity × unit_price) - discount_amount
);
```

#### الخطوة 4: إنشاء الدفعات (لكل طريقة دفع)
```sql
-- مثال: دفع مختلط (كاش + فيزا)
INSERT INTO payments (
  invoice_id,   -- من الخطوة 2
  account_id,   -- UUID حساب الكاش
  amount,       -- المبلغ المدفوع
  fee_amount,   -- 0 (الكاش بدون عمولة)
  net_amount    -- amount - fee_amount
);

INSERT INTO payments (
  invoice_id,   -- من الخطوة 2
  account_id,   -- UUID حساب الفيزا
  amount,       -- المبلغ المدفوع بالفيزا
  fee_amount,   -- amount × accounts.fee_percentage / 100
  net_amount    -- amount - fee_amount
);
```

#### الخطوة 5: تحديث المخزون
```sql
UPDATE products
SET stock_quantity = stock_quantity - :quantity,
    updated_at = now()
WHERE id = :product_id;

-- إذا stock_quantity <= min_stock_level بعد التحديث:
INSERT INTO notifications (user_id, type, title, body)
VALUES (admin_id, 'low_stock', 'مخزون منخفض', :product_name || ': ' || :remaining);
```

#### الخطوة 6: إنشاء قيود دفتر الأستاذ + تحديث أرصدة الحسابات
```sql
-- قيد لكل طريقة دفع
INSERT INTO ledger_entries (
  entry_date,      -- = invoice_date
  account_id,      -- نفس account_id من payments
  entry_type,      -- 'income'
  amount,          -- = payments.amount (المبلغ الإجمالي — Gross)
  reference_type,  -- 'invoice'
  reference_id,    -- = invoice_id
  description,     -- 'بيع فاتورة AYA-2026-00001'
  created_by       -- UUID المستخدم
);

-- إذا فيزا: قيد إضافي للعمولة
INSERT INTO ledger_entries (
  entry_date, account_id, entry_type, amount,
  reference_type, reference_id, description, created_by
) VALUES (
  invoice_date, visa_account_id, 'expense', fee_amount,
  'invoice', invoice_id, 'عمولة فيزا فاتورة AYA-2026-00001', user_id
);

-- ⚡ تحديث رصيد كل حساب مُستلم (C-01 Fix)
-- لكل payment في الفاتورة:
UPDATE accounts
SET current_balance = current_balance + :amount - :fee_amount
WHERE id = :account_id;
-- ملاحظة: UPDATE col = col + X هو atomic في PostgreSQL READ COMMITTED
-- في بيئة ≤5 مستخدمين لا حاجة لـ SELECT FOR UPDATE على accounts
```

#### الخطوة 7: تسجيل الدين (إذا وُجد)
```sql
-- فقط إذا debt_amount > 0
INSERT INTO debt_entries (
  debt_customer_id, invoice_id, amount, entry_date, notes, created_by
);

UPDATE debt_customers
SET current_balance = current_balance + :debt_amount,
    updated_at = now()
WHERE id = :debt_customer_id;

-- إذا current_balance > credit_limit → ERR_DEBT_LIMIT_WARNING (تنبيه فقط — لا يمنع البيع):
INSERT INTO notifications (user_id, type, title, body)
VALUES (admin_id, 'debt_limit_exceeded', 'تجاوز حد الدين', :customer_name || ': ' || :new_balance);
```

### 📊 مثال رقمي شامل

**سيناريو:** بيع iPhone (1 حبة × 450.000) + شاحن (2 × 8.000) + كفر (1 × 5.500)، خصم 5% على iPhone، دفع مختلط: 300 كاش + 121.500 فيزا (2% عمولة)، بدون دين.

#### حساب العناصر:

| المنتج | الكمية | السعر | الخصم% | الخصم | الإجمالي | التكلفة |
|--------|--------|-------|--------|-------|---------|---------|
| iPhone 15 | 1 | 450.000 | 5% | 22.500 | 427.500 | 400.000 |
| شاحن | 2 | 8.000 | 0% | 0 | 16.000 | 5.000 |
| كفر | 1 | 5.500 | 0% | 0 | 5.500 | 2.000 |

```
subtotal       = (1×450) + (2×8) + (1×5.5) = 471.500
discount_total = 22.500
total_amount   = 471.500 - 22.500 = 449.000
debt_amount    = 0
```

#### ما يُكتب في كل جدول:

**invoices** ← سجل واحد:
```
subtotal=471.500, discount_amount=22.500, total_amount=449.000, debt_amount=0, status='active'
```

**invoice_items** ← 3 سجلات:
```
{product=iPhone, qty=1, unit_price=450.000, cost_price_at_time=400.000, discount=22.500, total=427.500}
{product=شاحن,  qty=2, unit_price=8.000,   cost_price_at_time=5.000,   discount=0,      total=16.000}
{product=كفر,   qty=1, unit_price=5.500,    cost_price_at_time=2.000,   discount=0,       total=5.500}
```

**payments** ← سجلتان:
```
{account=الصندوق, amount=300.000, fee=0,     net=300.000}
{account=الفيزا,  amount=149.000, fee=2.980, net=146.020}
```

**products** ← 3 تحديثات:
```
iPhone: stock_quantity -= 1
شاحن:  stock_quantity -= 2
كفر:   stock_quantity -= 1
```

**ledger_entries** ← 3 قيود:
```
{account=الصندوق, type='income', amount=300.000, ref='invoice'}
{account=الفيزا,  type='income', amount=149.000, ref='invoice'}  ← المبلغ الإجمالي (Gross)
{account=الفيزا,  type='expense', amount=2.980,  ref='invoice'}  ← قيد العمولة
```

#### ✅ فحوصات التوازن:

```
✓ SUM(payments.amount) + debt_amount = total_amount
  → 300.000 + 149.000 + 0 = 449.000 ✓

✓ SUM(payments.amount) = SUM(ledger income)
  → 300.000 + 149.000 = 449.000, ledger income = 300.000 + 149.000 = 449.000 ✓

✓ الربح = SUM(qty × (unit_price - cost_price_at_time)) - discount
  → (1×(450-400)) + (2×(8-5)) + (1×(5.5-2)) - 22.500
  → 50.000 + 6.000 + 3.500 - 22.500 = 37.000 JOD ✓

✓ subtotal - discount = total
  → 471.500 - 22.500 = 449.000 ✓
```

|---------|--------|---------|-------|
| 1 | `invoices` | `SELECT` | قراءة الفاتورة الأصلية للتحقق |
| 2 | `invoice_items` | `SELECT` | قراءة العناصر للتحقق من الكميات |
| 3 | `returns` | `INSERT` | دائماً — سجل المرتجع الرئيسي |
| 4 | `return_items` | `INSERT` (عدة سجلات) | سجل لكل منتج مُرجع |
| 5 | `invoice_items` | `UPDATE` | تحديث `is_returned` + `returned_quantity` |
| 6 | `invoices` | `UPDATE status` | تغيير لـ `returned` أو `partially_returned` |
| 7 | `products` | `UPDATE stock_quantity` | إعادة الكمية للمخزون |
| 8 | `ledger_entries` | `INSERT` | قيد صادر (expense) من حساب الإرجاع |

### نوعان من المرتجع

| النوع | الوصف | تأثير على invoices.status |
|-------|-------|--------------------------|
| **مرتجع كامل (full)** | كل عناصر الفاتورة تُرجع | `'returned'` |
| **مرتجع جزئي (partial)** | بعض العناصر أو كمية أقل | `'partially_returned'` |

### التنفيذ خطوة بخطوة

#### الخطوة 1: التحقق (Validation)
```sql
-- قراءة الفاتورة الأصلية
SELECT id, status, total_amount, debt_customer_id, debt_amount
FROM invoices
WHERE id = :invoice_id;

-- التحقق 1: الفاتورة ليست مُلغاة
IF status = 'cancelled' → ERR_CANCEL_ALREADY + ROLLBACK

-- التحقق 2: الكمية المُرجعة لا تتجاوز المباعة
SELECT quantity, returned_quantity
FROM invoice_items WHERE id = :invoice_item_id;
IF return_quantity > (quantity - returned_quantity) → ERR_RETURN_QUANTITY + ROLLBACK

-- التحقق 3: idempotency
IF idempotency_key EXISTS → ROLLBACK (عملية مكررة)
```

#### الخطوة 2: إنشاء سجل المرتجع
```sql
INSERT INTO returns (
  return_number,         -- AYA-2026-00001 (auto-generated)
  original_invoice_id,   -- UUID الفاتورة الأصلية
  return_date,           -- CURRENT_DATE
  return_type,           -- 'full' أو 'partial'
  total_amount,          -- SUM(return_items.total_price)
  refund_account_id,     -- UUID اختياري: يُطلب فقط إذا وُجد cash_refund > 0
  reason,                -- سبب الإرجاع (إلزامي)
  created_by,            -- UUID المستخدم
  idempotency_key        -- UUID لمنع التكرار
) RETURNING id, return_number;
```

#### الخطوة 3: إنشاء عناصر المرتجع (لكل منتج)
```sql
INSERT INTO return_items (
  return_id,        -- من الخطوة 2
  invoice_item_id,  -- UUID عنصر الفاتورة الأصلية
  quantity,         -- الكمية المُرجعة
  unit_price,       -- = invoice_items.unit_price (سعر البيع الأصلي قبل الخصم، للتوثيق فقط)
  total_price       -- quantity × unit_price × (1 - discount_percentage / 100) (الصافي المُسترجع فعلياً)
);
```

#### الخطوة 4: تحديث الفاتورة الأصلية
```sql
-- تحديث عناصر الفاتورة
UPDATE invoice_items
SET returned_quantity = returned_quantity + :return_qty,
    is_returned = CASE
      WHEN returned_quantity + :return_qty >= quantity THEN true
      ELSE false
    END
WHERE id = :invoice_item_id;

-- تحديث حالة الفاتورة
UPDATE invoices
SET status = CASE
      WHEN :return_type = 'full' THEN 'returned'
      ELSE 'partially_returned'
    END,
    updated_at = now()
WHERE id = :invoice_id;
```

#### الخطوة 5: إعادة المخزون
```sql
UPDATE products
SET stock_quantity = stock_quantity + :returned_quantity,
    updated_at = now()
WHERE id = :product_id;
```

#### الخطوة 6: إنشاء قيود دفتر الأستاذ
```sql
-- قيد صادر (مبلغ الإرجاع يخرج من الحساب)
INSERT INTO ledger_entries (
  entry_date,      -- = return_date
  account_id,      -- = refund_account_id
  entry_type,      -- 'expense'
  amount,          -- المبلغ المُرجع
  reference_type,  -- 'return'
  reference_id,    -- = return_id
  description,     -- 'مرتجع فاتورة AYA-2026-00001'
  created_by
);
```

#### حالة خاصة: مرتجع فاتورة دين (تلقائي بالكامل)
```sql
-- return_total = إجمالي المرتجع (بعد الخصم)
-- remaining_debt = debt_entries.remaining_amount المرتبط بالفاتورة

debt_reduction = MIN(return_total, remaining_debt)
cash_refund    = return_total - debt_reduction

-- 1) تقليل الدين أولاً
UPDATE debt_entries
SET remaining_amount = remaining_amount - :debt_reduction,
    paid_amount      = amount - (remaining_amount - :debt_reduction),
    is_paid          = CASE WHEN remaining_amount - :debt_reduction = 0 THEN true ELSE false END
WHERE invoice_id = :invoice_id AND is_paid = false;

UPDATE debt_customers
SET current_balance = current_balance - :debt_reduction
WHERE id = :debt_customer_id;

-- 2) إنشاء قيد إرجاع نقدي فقط إذا كان هناك جزء مدفوع سابقاً
IF cash_refund > 0 THEN
  IF refund_account_id IS NULL THEN
    ERR_RETURN_REFUND_ACCOUNT_REQUIRED + ROLLBACK;
  END IF;

  INSERT INTO ledger_entries (
    entry_date, account_id, entry_type, amount,
    reference_type, reference_id, description, created_by
  ) VALUES (
    return_date, refund_account_id, 'expense', cash_refund,
    'return', return_id, 'إرجاع الجزء المسدد من فاتورة دين', user_id
  );
END IF;
```

### ⚠️ معالجة عمولة الفيزا عند المرتجع

```
┌──────────────────────────────────────────────────────────┐
│ 🔴 قاعدة مهمة: عمولة الفيزا لا تُستردّ!               │
│                                                          │
│ عند البيع: البنك خصم العمولة قبل ما يحوّل لنا.          │
│ يعني: دفع العميل 100 → وصلنا 98 (بعد 2% عمولة)         │
│                                                          │
│ عند المرتجع: نُرجع للعميل 100 من الصندوق (أو حسابه)    │
│ لكن لا نُرجع العمولة لأننا لم نستلمها أصلاً.           │
│                                                          │
│ Ledger: قيد الإرجاع = المبلغ الكامل 100                 │
│         (ليس net_amount 98)                               │
└──────────────────────────────────────────────────────────┘
```

**مثال:**
```
بيع أصلي بفيزا = 200.000 JOD
عمولة 2% = 4.000 → استلمنا 196.000

مرتجع:
→ نُرجع 200.000 للعميل (من الصندوق/كاش)
→ ledger_entry: expense, amount=200.000, account=الصندوق
→ الخسارة = 4.000 (العمولة اللي ما رجعت)
```

### 📊 مثال رقمي — مرتجع جزئي

**السيناريو:** من فاتورة OP-01 (449.000 JOD) → إرجاع iPhone فقط (427.500 بعد الخصم)، استرداد نقدي.

#### الفاتورة الأصلية:
```
iPhone:  qty=1, unit_price=450.000, discount=22.500, total=427.500
شاحن:   qty=2, unit_price=8.000,   discount=0,      total=16.000
كفر:    qty=1, unit_price=5.500,    discount=0,      total=5.500
إجمالي: 449.000
```

#### ما يُكتب في كل جدول:

**returns** ← سجل واحد:
```
return_number='AYA-2026-00100'
original_invoice_id=<invoice_id>
return_type='partial'
total_amount=427.500
refund_account_id=<cash_account_id>
reason='عيب تصنيعي'
```

**return_items** ← سجل واحد:
```
{invoice_item_id=<iphone_item_id>, quantity=1, unit_price=450.000, total_price=427.500}
```

> **ملاحظة:** `total_price = 427.500` (بعد الخصم) وليس `450.000` لأن العميل دفع المبلغ بعد الخصم.

**invoice_items** ← تحديث:
```
iPhone: returned_quantity = 0 → 1, is_returned = true
(شاحن وكفر: لا تتغير)
```

**invoices** ← تحديث:
```
status = 'active' → 'partially_returned'
```

**products** ← تحديث:
```
iPhone: stock_quantity += 1
```

**ledger_entries** ← قيد واحد:
```
{account=الصندوق, type='expense', amount=427.500, ref_type='return', ref_id=<return_id>}
```

#### ✅ فحوصات التوازن:

```
✓ SUM(return_items.total_price) = returns.total_amount
  → 427.500 = 427.500 ✓

✓ returned_quantity ≤ original quantity
  → 1 ≤ 1 ✓ (iPhone بِيعت 1 ورجعت 1)

✓ المخزون بعد الإرجاع = المخزون قبل + الكمية المُرجعة
  → stock_before + 1 = stock_after ✓

✓ الحساب بعد الإرجاع = الرصيد - مبلغ الإرجاع
  → cash_balance - 427.500 = new_balance ✓
```

## 🗂️ OP-03: ربط تعديل الفاتورة بالجداول (Edit Invoice → Tables)

> ⚠️ **أعقد عملية في النظام** — تستخدم نمط "عكس ثم إعادة تنفيذ" (Reverse + Reapply).

### المبدأ: لا تعدّل القيد — اعكسه وأنشئ جديد

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  المرحلة 1   │────▶│  المرحلة 2   │────▶│  المرحلة 3   │
│  التحقق      │     │  عكس القديم  │     │  تنفيذ الجديد│
└──────────────┘     └──────────────┘     └──────────────┘
```

### الجداول المتأثرة

| الترتيب | الجدول | المرحلة | العملية |
|---------|--------|---------|---------|
| 1 | `invoices` | تحقق | `SELECT` — قراءة الفاتورة الأصلية |
| 2 | `products` | تحقق | `SELECT FOR UPDATE` — قفل المنتجات |
| 3 | `ledger_entries` | عكس | `INSERT` — قيود عكسية (reversal) |
| 4 | `products` | عكس | `UPDATE` — إرجاع المخزون القديم |
| 5 | `debt_entries` + `debt_customers` | عكس | `UPDATE` — عكس الدين القديم (إن وُجد) |
| 6 | `invoice_items` | جديد | `DELETE` القديمة + `INSERT` الجديدة |
| 7 | `payments` | جديد | `DELETE` القديمة + `INSERT` الجديدة |
| 8 | `products` | جديد | `UPDATE` — خصم المخزون الجديد |
| 9 | `ledger_entries` | جديد | `INSERT` — قيود جديدة |
| 10 | `debt_entries` + `debt_customers` | جديد | `INSERT` + `UPDATE` (إن وُجد دين جديد) |
| 11 | `invoices` | تحديث | `UPDATE` — تحديث الإجماليات |
| 12 | `notifications` | إخطار | `INSERT` — إشعار لأحمد بالتعديل |

### التنفيذ خطوة بخطوة

#### المرحلة 1: التحقق (Validation)
```sql
-- قراءة الفاتورة الأصلية
SELECT * FROM invoices WHERE id = :invoice_id;

-- التحقق 1: الفاتورة ليست ملغاة
IF status = 'cancelled' → ERR_CANCEL_ALREADY + ROLLBACK

-- التحقق 2: لا يوجد مرتجع مرتبط
SELECT COUNT(*) FROM returns WHERE original_invoice_id = :invoice_id;
IF count > 0 → ERR_CANCEL_HAS_RETURN + ROLLBACK

-- التحقق 3: سبب التعديل إلزامي (حد أدنى من system_settings.require_reason_min_chars)
SELECT COALESCE((SELECT value FROM system_settings WHERE key = 'require_reason_min_chars'), '50')::INTEGER INTO v_min_chars;
IF LENGTH(edit_reason) < v_min_chars → ERR_CANCEL_REASON + ROLLBACK

-- التحقق 4: المخزون كافٍ للكميات الجديدة
-- (بعد إرجاع المخزون القديم في المرحلة 2)

-- التحقق 5: مجموع الدفعات = الإجمالي
IF SUM(new_payments.amount) + new_debt ≠ new_total → ERR_PAYMENT_MISMATCH + ROLLBACK
```

#### المرحلة 2: عكس العمليات القديمة (Reverse)
```sql
-- 2a: قيود عكسية في دفتر الأستاذ
-- لكل ledger_entry أصلي مرتبط بالفاتورة:
INSERT INTO ledger_entries (
  entry_date, account_id,
  entry_type,      -- عكس: income → expense, expense → income
  amount,          -- نفس المبلغ الأصلي
  reference_type,  -- 'reversal'
  reference_id,    -- = invoice_id
  description,     -- 'عكس تعديل فاتورة AYA-2026-00001 - ' + edit_reason
  created_by
);

-- 2b: إرجاع المخزون القديم
UPDATE products
SET stock_quantity = stock_quantity + :old_quantity
WHERE id = :old_product_id;

-- 2c: عكس الدين القديم (إن وُجد)
IF old_debt_amount > 0:
  UPDATE debt_customers
  SET current_balance = current_balance - :old_debt_amount
  WHERE id = :old_debt_customer_id;
  
  UPDATE debt_entries
  SET is_paid = true, paid_amount = amount, remaining_amount = 0
  WHERE invoice_id = :invoice_id;
```

#### المرحلة 3: تنفيذ العمليات الجديدة (Reapply)
```sql
-- 3a: حذف العناصر والدفعات القديمة
DELETE FROM invoice_items WHERE invoice_id = :invoice_id;
DELETE FROM payments WHERE invoice_id = :invoice_id;

-- 3b: إنشاء عناصر جديدة (مثل OP-01 خطوة 3)
INSERT INTO invoice_items (...) VALUES (...);

-- 3c: إنشاء دفعات جديدة (مثل OP-01 خطوة 4)
INSERT INTO payments (...) VALUES (...);

-- 3d: خصم المخزون الجديد
UPDATE products
SET stock_quantity = stock_quantity - :new_quantity
WHERE id = :new_product_id;

-- 3e: إنشاء قيود جديدة (مثل OP-01 خطوة 6)
INSERT INTO ledger_entries (...) VALUES (...);

-- 3f: تحديث الفاتورة بالإجماليات الجديدة
UPDATE invoices SET
  subtotal = :new_subtotal,
  discount_amount = :new_discount,
  total_amount = :new_total,
  debt_amount = :new_debt,
  debt_customer_id = :new_debt_customer_id,
  updated_at = now()
WHERE id = :invoice_id;

-- 3g: إشعار لأحمد بالتعديل
-- لا يوجد تنبيه retroactive_edit بعد اعتماد ADR-034 (No Backdating).
-- توثيق التعديل يتم في audit_logs فقط.
```

### 📊 مثال رقمي — تعديل فاتورة

**السيناريو:** تعديل فاتورة OP-01 — تبديل iPhone بـ Samsung (أرخص)، الشاحن والكفر يبقيان.

#### الفاتورة الأصلية (قبل التعديل):
```
iPhone:  qty=1, unit_price=450.000, discount=22.500, total=427.500, cost=400.000
شاحن:   qty=2, unit_price=8.000,   discount=0,      total=16.000,  cost=5.000
كفر:    qty=1, unit_price=5.500,    discount=0,      total=5.500,   cost=2.000
إجمالي: 449.000 (300 كاش + 149 فيزا)
```

#### الفاتورة الجديدة (بعد التعديل):
```
Samsung: qty=1, unit_price=350.000, discount=0,      total=350.000, cost=300.000
شاحن:   qty=2, unit_price=8.000,   discount=0,      total=16.000,  cost=5.000
كفر:    qty=1, unit_price=5.500,    discount=0,      total=5.500,   cost=2.000
إجمالي: 371.500 (371.500 كاش)
```

#### ما يحدث في كل مرحلة:

**المرحلة 2 — العكس:**
```
ledger_entries ← 3 قيود عكسية:
  {account=الصندوق, type='expense', amount=300.000, ref='reversal'} ← عكس وارد الكاش
  {account=الفيزا,  type='expense', amount=146.020, ref='reversal'} ← عكس وارد الفيزا
  {account=الفيزا,  type='income',  amount=2.980,   ref='reversal'} ← عكس عمولة الفيزا

products ← إرجاع المخزون:
  iPhone: stock += 1
  شاحن:  stock += 2
  كفر:   stock += 1
```

**المرحلة 3 — التنفيذ الجديد:**
```
invoice_items ← 3 سجلات جديدة:
  {Samsung, qty=1, price=350, cost=300, total=350}
  {شاحن,   qty=2, price=8,   cost=5,   total=16}
  {كفر,    qty=1, price=5.5,  cost=2,   total=5.5}

payments ← سجل واحد:
  {account=الصندوق, amount=371.500, fee=0, net=371.500}

products ← خصم جديد:
  Samsung: stock -= 1
  شاحن:   stock -= 2
  كفر:    stock -= 1

ledger_entries ← قيد جديد:
  {account=الصندوق, type='income', amount=371.500, ref='invoice'}

invoices ← تحديث:
  subtotal=371.500, discount=0, total=371.500, debt=0
```

#### ✅ فحوصات التوازن:

```
✓ عدد القيود العكسية = عدد القيود الأصلية
  → 3 عكسية = 3 أصلية ✓

✓ المخزون: إرجاع القديم ثم خصم الجديد
  → iPhone: +1 (رجع)، Samsung: -1 (بيع جديد) ✓

✓ SUM(new_payments) = new_total
  → 371.500 = 371.500 ✓

✓ الربح الجديد = SUM(qty × (price - cost)) - discount
  → (350-300) + (2×3) + (3.5) - 0 = 59.500 JOD ✓

✓ Ledger: SUM(income) - SUM(expense) = صافي الفاتورة الجديد
  → (reversals cancel out) + 371.500 = 371.500 ✓
```

## 🗂️ OP-04: ربط اللقطة اليومية بالجداول (Daily Snapshot → Tables)

> 📸 عملية **قراءة فقط** — تقرأ من عدة جداول وتحفظ ملخص في سجل واحد.

### الجداول المصدرية (للقراءة)

| الجدول | ما نقرأ منه |
|--------|------------|
| `invoices` | عدد الفواتير + إجمالي المبيعات (حسب `invoice_date`) |
| `returns` | عدد المرتجعات + إجمالي المرتجعات (حسب `return_date`) |
| `debt_entries` | إجمالي الدين المضاف خلال اليوم |
| `debt_payments` | إجمالي الدين المحصل خلال اليوم |
| `expenses` | إجمالي المصروفات (حسب `expense_date`) |
| `purchase_orders` | إجمالي المشتريات (حسب `purchase_date`) |
| `accounts` | أرصدة جميع الحسابات الحالية |

### الجداول المتأثرة (للكتابة)

| الجدول | العملية |
|--------|---------|
| `daily_snapshots` | `INSERT` — سجل واحد |

### التنفيذ خطوة بخطوة

#### الخطوة 1: حساب المبيعات
```sql
SELECT
  COUNT(*) AS invoice_count,
  COALESCE(SUM(total_amount), 0) AS total_sales
FROM invoices
WHERE invoice_date = CURRENT_DATE
  AND status != 'cancelled';
```

#### الخطوة 2: حساب المرتجعات
```sql
SELECT
  COUNT(*) AS return_count,
  COALESCE(SUM(total_amount), 0) AS total_returns
FROM returns
WHERE return_date = CURRENT_DATE;
```

#### الخطوة 3: حساب صافي المبيعات
```sql
net_sales = total_sales - total_returns
```

#### الخطوة 4: حساب حركة الديون
```sql
-- الدين المضاف (من فواتير بالدين + ديون يدوية)
SELECT COALESCE(SUM(amount), 0) AS total_debt_added
FROM debt_entries
WHERE DATE(created_at AT TIME ZONE 'Asia/Amman') = CURRENT_DATE;

-- الدين المحصل (تسديد الدين)
SELECT COALESCE(SUM(amount), 0) AS total_debt_collected
FROM debt_payments
WHERE payment_date = CURRENT_DATE;
```

#### الخطوة 5: حساب المصروفات والمشتريات
```sql
-- المصروفات
SELECT COALESCE(SUM(amount), 0) AS total_expenses
FROM expenses WHERE expense_date = CURRENT_DATE;

-- المشتريات
SELECT COALESCE(SUM(total_amount), 0) AS total_purchases
FROM purchase_orders WHERE purchase_date = CURRENT_DATE;
```

#### الخطوة 6: جلب أرصدة الحسابات
```sql
SELECT jsonb_agg(jsonb_build_object('account_id', id, 'name', name, 'balance', current_balance)) AS accounts_snapshot
FROM accounts;
```

#### الخطوة 7: إنشاء اللقطة
```sql
INSERT INTO daily_snapshots (
  snapshot_date,          -- CURRENT_DATE دائماً (ADR-034)
  total_sales,            -- من الخطوة 1
  total_returns,          -- من الخطوة 2
  net_sales,              -- من الخطوة 3
  invoice_count,          -- من الخطوة 1
  return_count,           -- من الخطوة 2
  total_debt_added,       -- من الخطوة 4
  total_debt_collected,   -- من الخطوة 4
  total_expenses,         -- من الخطوة 5
  total_purchases,        -- من الخطوة 5
  accounts_snapshot,      -- من الخطوة 6
  notes,                  -- اختياري
  created_by              -- UUID المستخدم (Admin)
) VALUES (
  CURRENT_DATE, :total_sales, :total_returns, :net_sales,
  :invoice_count, :return_count, :total_debt_added, :total_debt_collected,
  :total_expenses, :total_purchases, :accounts_snapshot,
  :notes, :admin_id
);
```

### 📌 ملاحظات مهمة

1. اللقطة اليومية **idempotent** بطبيعتها:
   - قيد `UNIQUE(snapshot_date)` يمنع إنشاء أكثر من لقطة لنفس اليوم.
   - إعادة الاستدعاء في نفس اليوم تُرجع نفس السجل ولا تنشئ نسخة جديدة.
2. لا يوجد Backdating مطلقاً (ADR-034):
   - `snapshot_date = CURRENT_DATE` فقط.
3. المنطقة الزمنية المعتمدة: `Asia/Amman` (ADR-037).

| 5 | `accounts` | `UPDATE` — current_balance (إن نقدي) |

### خطوات SOP 10/11 → SQL

```sql
-- SOP 10 خطوة 1-5: إنشاء أمر الشراء
INSERT INTO purchase_orders (
  purchase_number,     -- VARCHAR(20), auto, صيغة AYA-YYYY-NNNNN
  supplier_id,         -- UUID → suppliers(id), اختياري (SOP 10 خطوة 3)
  purchase_date,       -- DATE, default CURRENT_DATE
  total_amount,        -- DECIMAL(12,3), CHECK > 0
  is_paid,             -- BOOLEAN (ليس payment_type!)
                       --   true = نقدي (SOP 10)
                       --   false = آجل (SOP 11)
  payment_account_id,  -- UUID → accounts(id)
                       --   NOT NULL إن is_paid=true (SOP 10 خطوة 7)
                       --   NULL إن is_paid=false
  notes,               -- TEXT
  created_by           -- UUID → profiles(id)
) RETURNING id;

-- CHECK constraints من DB:
-- (is_paid = true AND payment_account_id IS NOT NULL)
--   OR (is_paid = false AND payment_account_id IS NULL)
-- (is_paid = true) OR (supplier_id IS NOT NULL)  ← المورد إلزامي إن آجل

-- SOP 10 خطوة 4-5: عناصر الشراء
INSERT INTO purchase_items (
  purchase_id,    -- UUID → purchase_orders(id) ON DELETE CASCADE
  product_id,     -- UUID → products(id) ON DELETE RESTRICT
  quantity,       -- INTEGER, CHECK > 0
  unit_cost,      -- DECIMAL(12,3), CHECK >= 0
  total_cost      -- DECIMAL(12,3)
);

-- SOP 10 خطوة 8: تحديث المخزون والتكلفة (DB عملية 7 خطوات 3-4)
UPDATE products SET
  stock_quantity = stock_quantity + :quantity,
  cost_price = :unit_cost,                -- آخر تكلفة
  avg_cost_price = (
    (avg_cost_price * :old_stock + :unit_cost * :quantity)
    / (NULLIF(:old_stock + :quantity, 0))
  ),
  updated_at = now()
WHERE id = :product_id;

-- SOP 10 خطوة 6-7: إن نقدي (is_paid=true) — DB عملية 7 خطوة 5
INSERT INTO ledger_entries (
  entry_date, account_id,
  entry_type,         -- 'expense'
  amount,             -- = total_amount
  reference_type,     -- 'purchase'
  reference_id,       -- = purchase_order_id
  description,        -- 'شراء بضاعة - أمر AYA-2026-00001'
  created_by
);

-- SOP 11: إن آجل (is_paid=false) — DB عملية 7 خطوة 6
UPDATE suppliers SET
  current_balance = current_balance + :total_amount,  -- دين علينا
  updated_at = now()
WHERE id = :supplier_id;

-- تحديث رصيد الحساب (إن نقدي)
UPDATE accounts SET
  current_balance = current_balance - :total_amount
WHERE id = :payment_account_id;

-- audit_logs — DB عملية 7 خطوة 7
INSERT INTO audit_logs (
  action_timestamp, user_id, action_type,  -- 'create_purchase'
  table_name,  -- 'purchase_orders'
  record_id, new_values, description, ip_address
);

-- حماية التكرار: UNIQUE(idempotency_key) WHERE NOT NULL (ADR-033) + UNIQUE(purchase_number)
-- idempotency_key إلزامي في API/RPC لعمليات الشراء، وpurchase_number يبقى معرفاً تجارياً فريداً
```

## 🗂️ OP-09: تسوية الحسابات (Reconciliation → Tables)

> ⚖️ **SOP 14** | **عملية DB 8: CreateReconciliation** | **دالة: `reconcile_account()`**

### الجداول المتأثرة

| # | الجدول | العملية |
|---|--------|---------|
| 1 | `ledger_entries` | `SELECT` — حساب الرصيد المتوقع |
| 2 | `reconciliation_entries` | `INSERT` — سجل التسوية |
| 3 | `ledger_entries` | `INSERT` — قيد تسوية (إن فرق ≠ 0) |
| 4 | `notifications` | `INSERT` — إشعار فرق التسوية |
| 5 | `audit_logs` | `INSERT` |

### خطوات SOP 14 → SQL

```sql
-- SOP 14 خطوة 1-3: حساب الرصيد المتوقع (DB عملية 8 خطوة 1)
SELECT
  SUM(CASE
    WHEN entry_type = 'income' THEN amount
    WHEN entry_type = 'expense' THEN -amount
    WHEN entry_type = 'adjustment' AND adjustment_direction = 'increase' THEN amount
    WHEN entry_type = 'adjustment' AND adjustment_direction = 'decrease' THEN -amount
  END) AS expected_balance
FROM ledger_entries
WHERE account_id = :account_id
  AND entry_date <= :reconciliation_date;

-- SOP 14 خطوة 4-5: إدخال الرصيد الفعلي وحساب الفرق
-- SOP 14 خطوة 6: إذا فرق → سبب إلزامي

-- ملاحظة تنفيذية: الإصدار الحالي من `reconcile_account()` لا يرمي
-- كود خطأ مخصص لحالة "تسوية مفتوحة لنفس اليوم".
-- يمكن تفعيل المنع الصارم لاحقاً عبر فهرس جزئي UNIQUE (انظر التوصية أدناه).

-- SOP 14 خطوة 7: إنشاء سجل التسوية (DB عملية 8 خطوة 2)
INSERT INTO reconciliation_entries (
  reconciliation_date,  -- DATE, default CURRENT_DATE
  account_id,           -- UUID → accounts(id) ON DELETE RESTRICT
  expected_balance,     -- DECIMAL(12,3)
  actual_balance,       -- DECIMAL(12,3), CHECK >= 0
  difference,           -- DECIMAL(12,3), CHECK = actual_balance - expected_balance
  difference_reason,    -- VARCHAR(255), Required ✅ (ليس notes!)
  is_resolved,          -- BOOLEAN:
                        --   true  ← إن difference = 0 (تسوية متطابقة)
                        --   false ← إن difference ≠ 0 (يحتاج معالجة)
  created_by            -- UUID → profiles(id)
);

-- إن فرق ≠ 0: قيد تسوية (DB عملية 8 خطوة 3)
INSERT INTO ledger_entries (
  entry_date, account_id,
  entry_type,              -- 'adjustment' (ليس income/expense!)
  amount,                  -- ABS(difference)
  adjustment_direction,    -- IF diff > 0 THEN 'increase' ELSE 'decrease'
  reference_type,          -- 'reconciliation'
  reference_id,            -- = reconciliation_entry_id
  description,             -- 'تسوية حساب الصندوق - نقص 15.000'
  created_by
);

-- إشعار فرق التسوية
INSERT INTO notifications (
  user_id, type,  -- 'reconciliation_difference' ✅ (من CHECK جدول 25)
  title, body, is_read, reference_type, reference_id
);

-- audit_logs (DB عملية 8 خطوة 4)
INSERT INTO audit_logs (
  action_timestamp, user_id, action_type,  -- 'reconcile_account'
  table_name,  -- 'reconciliation_entries'
  record_id, new_values, description, ip_address
);
```

### Function Signature
```
Input:  { account_id: UUID, actual_balance: DECIMAL, notes?: TEXT }
Output: { reconciliation_id: UUID, expected: DECIMAL, actual: DECIMAL, difference: DECIMAL }
Errors: ERR_UNAUTHORIZED
```

### 📊 مثال: نقص 15 JOD في الصندوق

| الجدول | البيانات |
|--------|---------|
| `reconciliation_entries` | expected=5000.000, actual=4985.000, difference=-15.000, difference_reason='نقص غير مبرر', **is_resolved=false** |
| `ledger_entries` | {الصندوق, **adjustment**, 15.000, adjustment_direction='**decrease**', ref='reconciliation'} |
| `notifications` | {type='**reconciliation_difference**', body='فرق -15.000 في الصندوق'} |
| `audit_logs` | {action_type='reconcile_account'} |

```
✓ SOP 14: "التسوية تُسجل فوراً وتدخل ضمن التقارير اليومية" ✓
✓ CHECK: difference = actual_balance - expected_balance → 4985 - 5000 = -15 ✓
```

### قاعدة `is_resolved`

| الحالة | القيمة | متى |
|--------|--------|-----|
| difference = 0 | `true` عند الإنشاء | تسوية متطابقة — لا حاجة لقيد تسوية |
| difference ≠ 0 | `false` عند الإنشاء | يحتاج معالجة |
| بعد المعالجة | `true` يدوياً | Admin فقط — بعد التحقيق وتغطية الفرق |

**آلية التحويل إلى true (يدوي):**
```sql
-- Admin فقط (RLS: UPDATE) بعد التحقيق في سبب الفرق
UPDATE reconciliation_entries
SET is_resolved = true
WHERE id = :reconciliation_id;

-- audit_logs إلزامي عند التحويل
INSERT INTO audit_logs (
  action_timestamp, user_id,
  action_type,   -- 'resolve_reconciliation'
  table_name,    -- 'reconciliation_entries'
  record_id,     -- = reconciliation_id
  old_values,    -- '{"is_resolved": false}'
  new_values,    -- '{"is_resolved": true}'
  description,   -- 'تم حل فرق التسوية -15.000 في الصندوق'
  ip_address
);
```

> ⚠️ **لا يوجد ربط تلقائي** في التصميم الحالي بين التسوية وقيود المعالجة.  
> التحويل يدوي بصلاحية Admin + audit_logs إلزامي.

### 🔒 توصية: فهرس UNIQUE جزئي (v2.3)

> **غير موجود حالياً في DB Design** — توصية لإصدار قادم.

```sql
-- يمنع أكثر من تسوية غير محلولة لنفس الحساب بنفس اليوم:
CREATE UNIQUE INDEX idx_reconciliation_unresolved_per_day
ON reconciliation_entries (account_id, reconciliation_date)
WHERE is_resolved = false;
```

| الحالة | السلوك |
|--------|--------|
| تسوية أولى (is_resolved=false) | ✅ يُسمح |
| تسوية ثانية نفس الحساب/اليوم (is_resolved=false) | ❌ DB error — UNIQUE violation |
| تسوية ثانية بعد حل الأولى (is_resolved=true) | ✅ يُسمح |

> **حالياً:** لا يوجد كود خطأ مخصص لهذه الحالة داخل RPC الحالي.  
> **مُقترح:** إضافة الفهرس كطبقة حماية ثانية على مستوى DB.

### 11c. تسديد دين (SOP 05 + DB عملية 4 + دالة `create_debt_payment()`)

```sql
-- SOP 05 خطوة 1-4: اختيار العميل + إدخال مبلغ + طريقة دفع

-- DB عملية 4 خطوة 1:
INSERT INTO debt_payments (
  debt_customer_id,   -- UUID → debt_customers(id)
  amount,             -- DECIMAL(12,3), CHECK > 0
  account_id,         -- UUID → accounts(id) — SOP 05 خطوة 4
  payment_date,       -- DATE, default CURRENT_DATE

  created_by,         -- UUID
  idempotency_key     -- UUID
);

-- DB عملية 4 خطوة 2: تطبيق FIFO على debt_entries
-- تسديد الأقدم أولاً (ORDER BY due_date ASC)
UPDATE debt_entries SET
  paid_amount = paid_amount + :applied_amount,
  remaining_amount = remaining_amount - :applied_amount,
  is_paid = CASE WHEN remaining_amount - :applied_amount = 0 THEN true ELSE false END,
  updated_at = now()
WHERE debt_customer_id = :debt_customer_id
  AND is_paid = false
ORDER BY due_date ASC;  -- FIFO

-- DB عملية 4 خطوة 2b: تسجيل التوزيع (لكل debt_entry تم تسديده)
INSERT INTO debt_payment_allocations (
  payment_id,         -- UUID → debt_payments(id)
  debt_entry_id,      -- UUID → debt_entries(id)
  allocated_amount    -- DECIMAL(12,3), CHECK > 0
);

-- DB عملية 4 خطوة 3:
UPDATE debt_customers SET
  current_balance = current_balance - :amount,
  updated_at = now()
WHERE id = :debt_customer_id;

-- DB عملية 4 خطوة 4:
INSERT INTO ledger_entries (
  entry_type='income', amount=:amount,
  reference_type='debt_payment', reference_id=:payment_id,
  account_id=:payment_account_id, ...
);

-- DB عملية 4 خطوة 5: audit_logs

  action_type='create_debt_payment', table_name='debt_payments', ...
);
```

### SOP 12 (جرد يومي) → SQL

```sql
-- SOP 12 خطوة 1: بدء عملية الجرد
INSERT INTO inventory_counts (
  count_date,      -- DATE, default CURRENT_DATE
  count_type,      -- ENUM: 'daily' (SOP 12) | 'weekly' | 'monthly' (SOP 13)
                   -- CHECK: IN ('daily', 'weekly', 'monthly')
  status,          -- ENUM: 'in_progress' (default)
                   -- CHECK: IN ('in_progress', 'completed')
  notes,           -- TEXT, optional
  created_by       -- UUID → profiles(id)
) RETURNING id;

-- SOP 12 خطوة 2-3: عد كل منتج
-- لكل منتج في المتجر:
SELECT stock_quantity AS system_quantity FROM products WHERE id = :product_id;

INSERT INTO inventory_count_items (
  inventory_count_id,  -- UUID → inventory_counts(id) ON DELETE CASCADE
  product_id,          -- UUID → products(id) ON DELETE RESTRICT
  system_quantity,     -- INTEGER = products.stock_quantity (وقت الجرد)
  actual_quantity,     -- INTEGER, CHECK >= 0 (ما يعده الموظف)
  difference,          -- INTEGER, CHECK = actual_quantity - system_quantity
  reason               -- VARCHAR(100), optional (SOP 12 خطوة 4 "سبب الفرق")
);

-- SOP 12 خطوة 5: إن وُجد فرق → تحديث المخزون
UPDATE products SET
  stock_quantity = :actual_quantity,
  updated_at = now()
WHERE id = :product_id AND :difference != 0;

-- SOP 12 خطوة 6: إكمال الجرد
-- إذا الجرد مكتمل مسبقاً → ERR_COUNT_ALREADY_COMPLETED
UPDATE inventory_counts SET
  status = 'completed',
  completed_at = now()
WHERE id = :count_id AND status = 'in_progress';

-- audit_logs
INSERT INTO audit_logs (
  action_timestamp, user_id,
  action_type,     -- 'complete_inventory_count'
  table_name,      -- 'inventory_counts'
  record_id,       -- = count_id
  new_values,      -- JSONB = {"count_type":"daily","items_counted":50,"differences":3}
  description,     -- 'جرد يومي — 3 فروقات من 50 منتج'
  ip_address
);
```

| الفرق | SOP 12 (يومي) | SOP 13 (شامل) |

| count_type | `'daily'` | `'monthly'` |
| النطاق | عينة سريعة | كل المنتجات |
| المستخدم | POS + Admin | Admin (SOP 13 خطوة 1) |
| RLS | POS: CREATE, UPDATE, SELECT | POS: CREATE, UPDATE, SELECT |

| الجدول | البيانات |

## OP-23: فحص انحراف الأرصدة (Balance Drift Check)

### الهدف
اكتشاف أي فرق بين `current_balance` المخزّن والرصيد المحسوب من المصدر.

### المستخدم
أحمد (Admin) — يدوياً أسبوعياً أو عند Daily Snapshot.

### السبب
ثلاثة جداول تحتوي أرصدة مُحدّثة يدوياً (denormalized):
- `accounts.current_balance` — يجب أن يساوي: افتتاحي + income - expense ± adjustment (من `ledger_entries`)
- `debt_customers.current_balance` — يجب أن يساوي: مجموع الديون غير المسددة (من `debt_entries`)
- `suppliers.current_balance` — يجب أن يساوي: مجموع المشتريات - مجموع التسديدات

أي خطأ في دالة أو transaction يمكن أن يُحدث drift بين المخزّن والمحسوب.

### قاعدة مصدر الحقيقة (Source of Truth)
- **المعتمد مالياً:** الرصيد المحسوب من الجداول المصدرية (`ledger_entries`, `debt_entries`, `supplier_payments`)
- **المخزّن (`current_balance`):** Cache تشغيلي للأداء فقط
- عند التعارض بين المخزّن والمحسوب: يُعتمد المحسوب وتُنفذ تسوية/إعادة احتساب

### الجداول المتأثرة
| الجدول | العمود | المصدر |
|--------|--------|--------|
| `accounts` | `current_balance` | `opening_balance` + `ledger_entries` |
| `debt_customers` | `current_balance` | `debt_entries` (غير مسددة) |
| `suppliers` | `current_balance` | `purchase_orders` - `supplier_payments` |

### المنطق (Pseudocode)

```sql
-- 1. فحص accounts
SELECT a.id, a.name, a.current_balance AS stored,
  a.opening_balance
    + COALESCE(SUM(CASE WHEN le.entry_type = 'income' THEN le.amount END), 0)
    - COALESCE(SUM(CASE WHEN le.entry_type = 'expense' THEN le.amount END), 0)
    + COALESCE(SUM(CASE WHEN le.entry_type = 'adjustment'
                    AND le.adjustment_direction = 'increase' THEN le.amount END), 0)
    - COALESCE(SUM(CASE WHEN le.entry_type = 'adjustment'
                    AND le.adjustment_direction = 'decrease' THEN le.amount END), 0)
  AS computed
FROM accounts a
LEFT JOIN ledger_entries le ON le.account_id = a.id
GROUP BY a.id
HAVING a.current_balance !=
  a.opening_balance
    + COALESCE(SUM(CASE WHEN le.entry_type = 'income' THEN le.amount END), 0)
    - COALESCE(SUM(CASE WHEN le.entry_type = 'expense' THEN le.amount END), 0)
    + COALESCE(SUM(CASE WHEN le.entry_type = 'adjustment'
                    AND le.adjustment_direction = 'increase' THEN le.amount END), 0)
    - COALESCE(SUM(CASE WHEN le.entry_type = 'adjustment'
                    AND le.adjustment_direction = 'decrease' THEN le.amount END), 0);

-- 2. فحص debt_customers
SELECT dc.id, dc.name, dc.current_balance AS stored,
  COALESCE(SUM(CASE WHEN de.is_paid = false THEN de.remaining_amount END), 0) AS computed
FROM debt_customers dc
LEFT JOIN debt_entries de ON de.debt_customer_id = dc.id
GROUP BY dc.id
HAVING dc.current_balance !=
  COALESCE(SUM(CASE WHEN de.is_paid = false THEN de.remaining_amount END), 0);

-- 3. فحص suppliers (بدون تضخيم جمع نتيجة JOIN)
SELECT
  s.id,
  s.name,
  s.current_balance AS stored,
  COALESCE((SELECT SUM(p.total_amount) FROM purchase_orders p WHERE p.supplier_id = s.id), 0)
  - COALESCE((SELECT SUM(sp.amount) FROM supplier_payments sp WHERE sp.supplier_id = s.id), 0)
  AS computed
FROM suppliers s
WHERE s.current_balance !=
  COALESCE((SELECT SUM(p.total_amount) FROM purchase_orders p WHERE p.supplier_id = s.id), 0)
  - COALESCE((SELECT SUM(sp.amount) FROM supplier_payments sp WHERE sp.supplier_id = s.id), 0);
```

### مثال drift حقيقي

**السيناريو:** فاتورة بيع مختلط (500 نقدي + 500 فيزا). Transaction فشل بعد تسجيل `ledger_entry` للنقدي لكن قبل تحديث `accounts.current_balance`:

| الكيان | stored | computed | drift |
|--------|--------|----------|-------|
| الصندوق (نقدي) | 10,000 | 10,500 | **+500** |

**السبب:** `ledger_entry` سُجّل لكن `UPDATE accounts SET current_balance` لم يُنفّذ (timeout).

**المعالجة:**
1. Admin يرى التنبيه
2. يراجع آخر `ledger_entries` للحساب
3. يُجري تسوية (OP-14) إذا تأكد من صحة الـ ledger

### السلوك عند وجود drift

| الحالة | الإجراء |
|--------|---------|
| drift = 0 لجميع الكيانات | ✅ لا إجراء — سجل نتيجة |
| drift ≠ 0 لأي كيان | 🔴 إشعار فوري لأحمد + تسجيل في `audit_logs` |

### الصلاحيات
- **Admin:** تشغيل الفحص + رؤية النتائج
- **POS:** لا وصول

## 📋 صفحة قرار: إلغاء vs إرجاع vs تعديل

### متى تستخدم كل خيار؟

| السؤال | الإجابة | الإجراء |
|--------|---------|---------|
| الفاتورة خطأ بالكامل (عميل خاطئ، كل المنتجات خطأ)؟ | نعم | **إلغاء (OP-10)** |
| العميل أرجع بعض المنتجات؟ | نعم | **إرجاع جزئي (OP-04)** |
| العميل أرجع كل المنتجات؟ | نعم | **إرجاع كامل (OP-04)** — لا إلغاء |
| تريد تغيير طريقة الدفع فقط؟ | نعم | **تعديل (OP-11)** — إن كان مسموحاً |
| الفاتورة بها مرتجع مسبق؟ | نعم | **لا يمكن إلغاء** — استخدم إرجاع للباقي |
| الفاتورة ملغاة مسبقاً؟ | نعم | **لا إجراء** — أنشئ فاتورة جديدة |

### قواعد صارمة

| القاعدة | التفسير |
|---------|---------|
| الإلغاء يعكس كل شيء | مخزون + مالية + ديون (لو آجل) |
| الإلغاء يتطلب سبب | `cancellation_reason` إلزامي |
| الإرجاع لا يُلغى | إرجاع خاطئ = فاتورة جديدة |
| التعديل محدود | لا تغيير كميات — فقط metadata (ملاحظات، طريقة دفع إذا لم يُقفل اليوم) |

## OP-24: الإقفال اليومي (Daily Close)

### الهدف
قائمة مراجعة نهاية اليوم تربط Snapshot + Drift Check + التسوية.

### المستخدم
أحمد (Admin) — نهاية كل يوم عمل.

### الخطوات

| الخطوة | الإجراء | المرجع |
|--------|---------|--------|
| 1 | مراجعة فواتير اليوم — هل يوجد فواتير draft/معلّقة؟ | شاشة الفواتير |
| 2 | تسوية حساب الصندوق (عد الكاش الفعلي) | OP-14 / SOP-14 |
| 3 | تشغيل Daily Snapshot | OP-17 / SOP-18 |
| 4 | تشغيل Balance Drift Check | OP-23 / SOP-24 |
| 5 | مراجعة الإشعارات — هل يوجد تنبيهات لم تُعالج؟ | NotificationBell |
| 6 | تأكيد: "اليوم مُقفل" | - |

### السلوك
- **ليس إقفال صلب (Hard Close)** — النظام لا يمنع العمليات بعد الإقفال
- هو **soft checklist** للتأكد من سلامة البيانات
- إذا drift ≠ 0 → لا تُقفل حتى تُحل التسوية

### البيانات المتأثرة
| الجدول | الإجراء |
|--------|---------|
| `daily_snapshots` | INSERT (لقطة اليوم) |
| `reconciliation_entries` | INSERT (إن وجدت تسوية) |
| `audit_logs` | INSERT (تسجيل الإقفال) |

## 📊 Failure Modes & Rollback Boundaries (مرجع شامل)

### جدول أنماط الفشل لكل عملية

| العملية | نقطة الفشل المحتملة | Rollback تلقائي؟ | الإجراء اليدوي |
|---------|---------------------|-----------------|---------------|
| **OP-01 بيع** | بعد ledger_entry وقبل inventory update | ✅ Transaction rollback | لا إجراء — كل شيء يعود |
| **OP-01 بيع** | بعد commit — اكتشاف خطأ لاحقاً | ❌ لا | إلغاء (OP-10) أو إرجاع (OP-04) |
| **OP-04 إرجاع** | بعد return وقبل ledger refund | ✅ Transaction rollback | لا إجراء |
| **OP-04 إرجاع** | بعد commit — إرجاع خاطئ | ❌ لا | فاتورة بيع جديدة |
| **OP-07 تسديد دين** | بعد payment وقبل balance update | ✅ Transaction rollback | لا إجراء |
| **OP-10 إلغاء** | بعد commit — إلغاء خاطئ | ❌ لا | فاتورة جديدة |
| **OP-14 تسوية** | خطأ حسابي في الفرق | ❌ لا | تسوية جديدة عكسية |
| **OP-17 Snapshot** | Snapshot يُنشأ بأرصدة خاطئة | ❌ لا | drift check يكشف |
| **OP-23 Drift** | Drift detected | ❌ لا | تسوية (OP-14) |
| **تسجيل مصروف** | بعد commit — مصروف خاطئ | ❌ لا | مصروف عكسي (reversal) |

### القاعدة العامة

> - **قبل COMMIT:** كل شيء يعود تلقائياً (PostgreSQL transaction)
> - **بعد COMMIT:** لا rollback — يُنشأ عملية تصحيحية (إلغاء / إرجاع / تسوية)
> - **audit_logs:** لا تُعدّل ولا تُحذف أبداً (ADR-039)

### ملاحظة حول Concurrency (C-05)

> `UPDATE accounts SET current_balance = current_balance ± X` هو **atomic** في PostgreSQL READ COMMITTED.  
> في بيئة ≤5 مستخدمين مع عمليات متباعدة، لا حاجة لـ `SELECT FOR UPDATE` على `accounts`.  
> إذا توسع النظام لعملاء خارجيين: يتطلب Advisory Locks أو `SELECT FOR UPDATE`.

## 🗂️ OP-22: تعديل إعدادات النظام (Update Settings → Tables)

> **الصلاحية:** Admin فقط

### الجداول المتأثرة

| # | الجدول | العملية |
|---|--------|---------|
| 1 | `system_settings` | `UPDATE value, updated_at, updated_by` |
| 2 | `audit_logs` | `INSERT` — إلزامي مع old_values + new_values |

### التنفيذ خطوة بخطوة

```sql
-- ═══ المرحلة 1: التحقق ═══

-- التحقق 1: الصلاحية
IF auth.role != 'admin' → ERR_UNAUTHORIZED + ROLLBACK

-- التحقق 2: المفتاح موجود
SELECT id, value AS old_value, value_type FROM system_settings WHERE key = :key;
IF NOT FOUND → ERR_SETTING_NOT_FOUND + ROLLBACK

-- التحقق 3: نوع القيمة
IF value_type = 'boolean' AND :new_value NOT IN ('true', 'false') → ERR_VALIDATION_INCORRECT_TYPE
IF value_type = 'number' AND :new_value !~ '^[0-9]+$' → ERR_VALIDATION_INCORRECT_TYPE

-- التحقق 4: النطاق (لكل مفتاح حسب Validation Rules)
-- مثال: max_pos_discount_percentage → 0 ≤ value ≤ 100
IF :key = 'max_pos_discount_percentage' AND (:new_value < 0 OR :new_value > 100)
  → ERR_VALIDATION_OUT_OF_RANGE + ROLLBACK

-- ═══ المرحلة 2: التنفيذ ═══

UPDATE system_settings SET
  value = :new_value,
  updated_at = now(),
  updated_by = auth.uid()
WHERE key = :key;

-- ═══ المرحلة 3: التسجيل ═══

INSERT INTO audit_logs (
  action_timestamp, user_id, action_type,
  table_name, record_id,
  old_values, new_values,
  description
) VALUES (
  now(), auth.uid(), 'update_setting',
  'system_settings', :setting_id,
  jsonb_build_object(:key, :old_value),
  jsonb_build_object(:key, :new_value),
  'تعديل إعداد ' || :key || ': ' || :old_value || ' → ' || :new_value
);
```

### Function Signature
```
Input:  { updates: [{ key: VARCHAR, value: VARCHAR }] }
Output: { success: BOOLEAN, updated_keys: INT }
Errors: ERR_UNAUTHORIZED, ERR_SETTING_NOT_FOUND, ERR_VALIDATION_INCORRECT_TYPE, ERR_VALIDATION_OUT_OF_RANGE
```

> **ملاحظة تزامن:** القيمة المحدّثة تصبح سارية فوراً لأي Transaction جديد. العمليات الجارية (مثل بيع قيد التنفيذ) تستخدم القيمة التي قرأتها داخل الـ Transaction الخاص بها.
