# برومبت لحل 3 فجوات توثيقية — نظام آية موبايل

---

## السياق العام

أنت تعمل على نظام "آية موبايل" — نظام نقطة بيع (POS) + إدارة مخزون + ديون + محاسبة لمتجر هواتف صغير في الأردن.

**المستخدمون:** 2 فقط (أحمد = المالك/Admin، موظف POS)
**التقنيات:** Supabase (PostgreSQL + Auth + RLS) + Next.js 14 (App Router) + Vercel
**العملة:** دينار أردني (JOD) — `د.أ`
**اللغة:** عربية (RTL)
**الاتصال:** Online-only (لا يوجد وضع Offline)

**الملفات الموجودة:** 28 ملف توثيق في `aya-mobile-documentation/` + 5 ملفات SQL في `supabase/migrations/`

---

## المطلوب منك

أنت مطلوب منك حل **3 فجوات توثيقية** بإنشاء/تعديل ملفات في مجلد `aya-mobile-documentation/`. لا تعدل أي ملف SQL. لا تنشئ كود تطبيق.

اقرأ كل ملف مشار إليه **بالكامل** قبل أي تعديل. لا تفترض — تحقق.

## الفجوة 1: ملف مرجعي للبناء — `28_Reference_Implementation.md`

### المشكلة
التوثيق الحالي يشرح **ماذا** يُبنى لكن لا يوجد مثال واحد كامل يوضح **كيف** يُكتب الكود. هذا يعني أن AI أو مطور سيكتب كل route بطريقة مختلفة وقد يخالف قواعد الأمان.

### المطلوب
أنشئ ملف `aya-mobile-documentation/28_Reference_Implementation.md` يحتوي على:

#### القسم 1: النمط الإلزامي لكل API Route (Canonical API Route Pattern)
اكتب مثال كامل لـ `POST /api/sales` (route.ts) يشمل بالضبط:

1. **استيراد عميل Supabase** — من `@/lib/supabase/admin` (service_role)
2. **استخراج الجلسة والتحقق منها** — من `@supabase/ssr` عبر cookies
3. **التحقق من الصلاحيات** (role check) — admin أو pos_staff حسب العملية
4. **Zod Schema** كامل للتحقق من بيانات الطلب مع كل الحقول المذكورة في `25_API_Contracts.md` لعملية البيع
5. **استدعاء RPC** — `supabaseAdmin.rpc('create_sale', { ... })`
6. **معالجة الأخطاء** — تحويل أخطاء PostgreSQL (`RAISE EXCEPTION 'ERR_*'`) إلى response JSON بصيغة StandardEnvelope:
```json
{ "success": false, "error": { "code": "ERR_STOCK_INSUFFICIENT", "message": "المخزون غير كافٍ لهذا المنتج" } }
```
7. **صيغة النجاح** — StandardEnvelope:
```json
{ "success": true, "data": { ... } }
```

استخرج رسائل الخطأ العربية من `16_Error_Codes.md` — عمود "رسالة المستخدم (AR)".

**مراجع إلزامية تقرأها قبل الكتابة:**
- `13_Tech_Config.md` — القواعد الست الإلزامية (سطر 110-116) + هيكل المجلدات (سطر 304+)
- `25_API_Contracts.md` — عقود API (اقرأ `POST /api/sales` كمرجع)
- `16_Error_Codes.md` — جميع أكواد الخطأ ورسائلها العربية
- `10_ADRs.md` — خصوصاً ADR-042 (API-First)، ADR-043 (Server-Authoritative)، ADR-044 (Revoke-All)
- `15_Seed_Data_Functions.md` — توقيع دالة `create_sale()` (القسم 1)

#### القسم 2: النمط الإلزامي للقراءة (Canonical Read Pattern)
اكتب مثال كامل لقراءة قائمة المنتجات من المتصفح:
1. استخدام `@supabase/ssr` مع `anon_key` (ليس service_role)
2. RLS تحمي البيانات تلقائياً
3. Blind POS — لا يقرأ POS أعمدة `cost_price` أو `current_balance`
4. استخدام SWR أو React Query للكاش

**مرجع:** `13_Tech_Config.md` سطر 176-198 (POS Performance Guidelines + Product Pre-loading)

#### القسم 3: خريطة تحويل الأخطاء (Error Mapping Table)
جدول كامل يربط كل `ERR_*` بـ:
- HTTP status code
- رسالة المستخدم بالعربية (من `16_Error_Codes.md`)
- مثال كود TypeScript لتحويل خطأ Supabase RPC إلى StandardEnvelope

**مرجع:** `16_Error_Codes.md` — اقرأه بالكامل

#### القسم 4: قواعد إلزامية لكل مطور/AI
اكتب قائمة "يجب" و"يُمنع" واضحة:

**يجب:**
- كل API Route يتبع نفس النمط (Session → Role → Zod → RPC → Response)
- كل كتابة تمر عبر API Route (لا INSERT مباشر من المتصفح)
- السعر يُسحب من DB وليس من المتصفح (ADR-043)
- `idempotency_key` يُولّد في الواجهة كـ UUID v4

**يُمنع:**
- استخدام `service_role` في أي كود يعمل في المتصفح
- قراءة `cost_price` أو `current_balance` من POS مباشرة
- قبول `unit_price` أو `total_amount` من الواجهة في عمليات البيع
- تعديل أو حذف `ledger_entries` أو `audit_logs`

#### القسم 5: أنماط الواجهة (UI Patterns)
اكتب مثال TypeScript/React لمكون واجهة يستدعي API Route ويعالج النتيجة:
- زر "إتمام البيع" مع تعطيل أثناء الإرسال
- عرض Toast عربي عند النجاح (sonner)
- عرض Toast خطأ عربي عند الفشل
- حفظ السلة في zustand مع persist

**مرجع:** `13_Tech_Config.md` سطر 200-270 (Optimistic UI + Post-Sale + Loading States)

#### التنسيق
- اتبع نفس تنسيق بقية ملفات التوثيق (عنوان + جدول إصدارات + روابط ملفات مرتبطة)
- أضف في النهاية:
```
**الإصدار:** 1.0
**تاريخ التحديث:** 5 مارس 2026
```

## الفجوة 2: إدارة المستخدمين والأجهزة — تحديث `02_Gaps_Risks_Recommendations.md` + إضافة قسم في `08_SOPs.md`

### المشكلة
لا يوجد توثيق لسيناريوهات حقيقية: نسيان كلمة المرور، فقدان جهاز، تغيير صلاحيات، إنهاء جلسات.

### المطلوب — الجزء أ: تحديث `02_Gaps_Risks_Recommendations.md`

اقرأ الملف بالكامل أولاً. ثم:

1. أضف فجوة جديدة في قسم "فجوات حرجة":

```
| GAP-07 | **إدارة المستخدمين والأجهزة** غير موثقة: نسيان كلمة المرور، فقدان جهاز، تغيير صلاحيات، إنهاء جلسات | `08_SOPs.md`, `10_ADRs.md` | إضافة SOPs لإدارة المستخدمين (SOP-20 إلى SOP-24) | MVP |
```

2. أضف خطر جديد في "مخاطر تقنية":

```
| RISK-14 | **جهاز مفقود/مسروق فيه جلسة نشطة** — المهاجم يمكنه استخدام النظام | 🔴 | إنهاء جميع جلسات المستخدم عبر Supabase Auth Admin API + تغيير كلمة المرور فوراً |
```

3. حدّث ملخص المخاطر ليشمل RISK-14.
4. حدّث الإصدار والتاريخ.

### المطلوب — الجزء ب: إضافة SOPs في `08_SOPs.md`

اقرأ الملف بالكامل لفهم تنسيق SOPs الحالية. ثم أضف في نهاية الملف (قبل الإصدار):

**SOP-20: إنشاء مستخدم جديد (Admin فقط)**
خطوات:
1. الدخول لـ Supabase Dashboard → Authentication → Users
2. إنشاء مستخدم بـ email + password
3. الانتظار حتى يظهر في جدول `profiles` (trigger تلقائي)
4. تحديث `role` في `profiles` (admin أو pos_staff)
5. إعطاء الموظف كلمة المرور + رابط النظام
6. الطلب من الموظف تغيير كلمة المرور فوراً

**SOP-21: إعادة تعيين كلمة مرور منسية (Admin فقط)**
خطوات:
1. الدخول لـ Supabase Dashboard → Authentication → Users
2. اختيار المستخدم → Send Password Reset Email
3. أو: تحديث كلمة المرور يدوياً من Dashboard
4. إبلاغ الموظف بكلمة المرور الجديدة
5. الطلب من الموظف تغييرها فوراً

**SOP-22: التعامل مع جهاز مفقود/مسروق (Admin فقط — طوارئ)**
خطوات:
1. **فوراً:** الدخول لـ Supabase Dashboard → Authentication → Users
2. اختيار المستخدم المتأثر
3. تغيير كلمة المرور (هذا يُبطل الجلسات تلقائياً في Supabase)
4. **اختياري:** تعطيل الحساب مؤقتاً (`is_active = false` في profiles)
5. التأكد من عدم وجود عمليات مشبوهة في `audit_logs`
6. بعد تأمين جهاز جديد: إعادة تفعيل الحساب + كلمة مرور جديدة

**SOP-23: تغيير صلاحيات مستخدم (Admin فقط)**
خطوات:
1. التأكد من عدم وجود جلسة نشطة للمستخدم (أو إعادة تعيين كلمة المرور لإنهائها)
2. تحديث `role` في جدول `profiles` عبر Supabase Dashboard → Table Editor
3. التحقق: تسجيل الدخول بحساب المستخدم والتأكد من الصلاحيات الجديدة
4. تسجيل التغيير في `audit_logs` يدوياً إذا لم يكن تلقائياً

**SOP-24: إنهاء جميع الجلسات النشطة (Admin فقط — طوارئ)**
خطوات:
1. الدخول لـ Supabase Dashboard → Authentication → Users
2. اختيار المستخدم
3. تغيير كلمة المرور — هذا يُبطل كل الـ refresh tokens
4. أو: استخدام Supabase Admin API: `supabase.auth.admin.deleteUser()` ثم إعادة إنشائه (حل أقصى)
5. المستخدم سيُطلب منه تسجيل الدخول مجدداً من كل جهاز

**التنسيق:** اتبع نفس تنسيق SOPs الموجودة (عنوان + الهدف + المستخدم + المتطلبات + جدول خطوات + أخطاء شائعة). حدّث الإصدار.

## الفجوة 3: فحص السلامة المالية في الواجهة — تحديث `09_Implementation_Plan.md` + `03_UI_UX_Sitemap.md`

### المشكلة
دالة `check_balance_drift()` موجودة في SQL لكن لا يوجد في التوثيق:
- شاشة تعرض نتيجة الفحص
- آلية تنبيه يومي
- ماذا يحدث لو اكتشف انحراف

### المطلوب — الجزء أ: تحديث `09_Implementation_Plan.md`

اقرأ الملف بالكامل. في قسم MVP (المرحلة 1)، تحت "اللقطة اليومية والتقارير بالتاريخ (Week 5)"، أضف صف جديد:

```
| فحص السلامة المالية | زر "فحص الأرصدة" في Dashboard يستدعي `check_balance_drift()` ويعرض النتيجة | يظهر ✅ إذا لا انحراف، أو 🔴 مع تفاصيل الانحراف |
```

وفي قسم V1 (المرحلة 2)، أضف:

```
| تنبيه سلامة يومي تلقائي | Cron Job يستدعي `check_balance_drift()` يومياً الساعة 00:01 + إشعار Admin إذا اكتشف انحراف | الإشعار يصل في `notifications` |
```

### المطلوب — الجزء ب: تحديث `03_UI_UX_Sitemap.md`

اقرأ الملف بالكامل لفهم التنسيق. أضف في شاشة Dashboard (الرئيسية) مكون جديد:

**بطاقة "سلامة الأرصدة" (Balance Integrity Card)**

```
┌─────────────────────────────────────┐
│  🛡️ سلامة الأرصدة                   │
│                                     │
│  ✅ جميع الأرصدة متطابقة            │
│  آخر فحص: اليوم 14:30               │
│                                     │
│  [🔄 فحص الآن]                      │
└─────────────────────────────────────┘
```

حالة الانحراف:
```
┌─────────────────────────────────────┐
│  🛡️ سلامة الأرصدة                   │
│                                     │
│  🔴 انحراف في 2 حساب                │
│  الصندوق: فرق 5.000 د.أ             │
│  فيزا: فرق -2.500 د.أ               │
│                                     │
│  [📋 عرض التفاصيل] [🔄 فحص مجدداً] │
└─────────────────────────────────────┘
```

- الصلاحية: Admin فقط (POS لا يرى هذه البطاقة)
- الزر يستدعي `POST /api/health/balance-check` → `check_balance_drift()`
- النتيجة تُخزن مؤقتاً في الواجهة (لا إعادة فحص تلقائي كل ثانية)

أضف أيضاً API Route في هيكل المجلدات في `13_Tech_Config.md`:
```
│   │   ├── health/
│   │   │   ├── route.ts              # GET health check
│   │   │   └── balance-check/route.ts # POST check_balance_drift (Admin)
```

حدّث الإصدارات والتواريخ لكل ملف تعدله.

## قواعد التنفيذ الإلزامية

1. **اقرأ كل ملف مشار إليه بالكامل قبل أي تعديل** — لا تفترض محتواه
2. **لا تكرر معلومات موجودة** — أشر إليها بالرابط
3. **اتبع نفس تنسيق الملفات الموجودة** — نفس العناوين، الجداول، الأسلوب
4. **لا تعدل ملفات SQL** — التعديلات على التوثيق فقط
5. **لا تنشئ كود تطبيق** — فقط أمثلة مرجعية داخل التوثيق
6. **كل مثال كود يجب أن يكون TypeScript** واقعي وقابل للنسخ
7. **استخدم المكتبات المعتمدة فقط** من `13_Tech_Config.md`: `@supabase/supabase-js`, `@supabase/ssr`, `zod`, `zustand`, `sonner`, `lucide-react`
8. **الأمثلة تستخدم App Router** (Next.js 14) — ليس Pages Router
