# تقرير المالك — تحليل نظام الموارد البشرية

---

## أ. ماذا يرى المدير الآن عند فتح قسم الموارد البشرية؟

تخيل أنك مدير تدخل إلى النظام صباحاً وتريد متابعة موظف:

1. **تفتح قائمة الموارد البشرية** — تجد: الموظفين، الإجازات، الرواتب، الحضور، العهد، السلف
2. **تدخل إلى "الموظفين"** — ترى قائمة بالموظفين مع فلاتر للقسم والحالة والنوع
3. **تضغط على موظف** — تفتح صفحة الملف الشخصي:
   - venster صغير يعرض: الاسم، الهاتف، الحالة، نوع الموظف، الراتب، طريقة الدفع، IBAN
   - ثلاث بانرات كبيرة تأخذ 40% من الشاشة
   - روابط لـ: الإجازات، الحضور، الرواتب، العهد، السلف
   - **المشكلة**: لا ترى أي بيانات فعلية هنا — فقط روابط

4. **تضغط على "الحضور"** — الصفحة تفتح **كل موظفي الشركة** بدلاً من الموظف الذي كنت تنظر إليه
5. **تضغط على "الرواتب"** — نفس المشكلة، تفقد سياق الموظف
6. **تريد إعدادات HR** — **لا تجدها!** الإعدادات مخفية عنك كمدير

**الخلاصة**: النظام يعمل كـ "مجموعة أدوات منفصلة" — كل صفحة معزولة عن الأخرى.

## ب. ما الذي يبدو أنه يعمل لكنه في الحقيقة مكسور؟

### المشكلة 1: راتب المندوب الداخلي قد يُحسب بطريقة خاطئة

**المثال العملي:**
- عندك مندوب داخلي اسمه "أحمد"
- في شاشة الرواتب، يظهر أن نوعه "راتب ثابت"
- لكن في قاعدة البيانات، النظام سجله "عمولة"
- **النتيجة**: راتب أحمد يُحسب بنظام العمولات بدلاً من الراتب الثابت

**لماذا هذا خطير؟**
- إذا كان راتب أحمد 500 دينار ثابتة، والنظام يحسبه عمولة
- قد يحصل أحمد على مبلغ مختلف كل شهر بدون سبب واضح
- قد تظهر لك أرقام غير صحيحة في كشوف الرواتب

### المشكلة 2: فقدان السياق بين الصفحات

**المثال العملي:**
- فتحت ملف "محمد" الموظف
- رأيت أن عنده 3 أيام غياب
- ضغطت على "الحضور" لتسجيل استثناء
- **فوجئت أنك في صفحة حضور كل الموظفين**
- اضطررت تبحث عن محمد مرة ثانية

**النتيجة**: كل مرة تنتقل بين الصفحات، تضيع 30-60 ثانية في البحث من جديد.

### المشكلة 3: صفحة "لوحة التحكم" موجودة لكن لا أحد يراها

**المثال العملي:**
- هناك صفحة كاملة (225 سطر برمجة) تعرض ملخص HR
- ليس هناك أي رابط لها في القائمة
- يجب كتابة الرابط يدوياً للوصول إليها

## ج. ما هي المشكلة الرئيسية في شكل القسم؟

### الفرق بين "مجموعة أدوات منفصلة" و"نظام موارد بشرية حقيقي"

| مجموعة أدوات منفصلة | نظام HR حقيقي |
|---------------------|---------------|
| كل صفحة تعمل لوحدها | كل شيء مترابط حول الموظف |
| تفقد سياق الموظف عند الانتقال | السياق يبقى محفوظاً دائماً |
| تبحث عن نفس الموظف في كل صفحة | تفتح ملف الموظف مرة واحدة |
| روابط فقط — لا بيانات | بيانات فعلية في كل مكان |

### ماذا يعني "ملف الموظف كمركز القيادة"؟

**الوضع الحالي:**
```
ملف محمد ← رابط → الإجازات ← بحث عن محمد ← عرض إجازات محمد
```

**الوضع المطلوب:**
```
ملف محمد ← تبويب → الإجازات (تظهر مباشرة)
ملف محمد ← تبويب → الحضور (يظهر مباشرة)
ملف محمد ← تبويب → الرواتب (يظهر مباشرة)
```

**المثال الملموس:**
تخيل أنك تفتح ملف موظف في نظام HR متكامل:
- في الأعلى: بيانات الموظف الأساسية
- تبويبات: ملخص | مستندات | إجازات | حضور | رواتب | عهد | سلف
- عند الضغط على أي تبويب — البيانات تظهر **في نفس الصفحة**
- لا تحتاج للبحث، لا تفقد السياق

## د. كيف سيبدو النظام بعد الإصلاح — للمناديب الداخليين؟

**من هو المندوب الداخلي؟**
- موظف يعمل داخل المستودع
- يتقاضى راتباً (ثابتاً أو عمولة)
- **لا يظهر** في قائمة مناديب العمليات
- له محفظة مالية داخلية

**التجربة الجديدة:**

1. **في قائمة الموظفين**:
   - يظهر بعلامة "مندوب داخلي"
   - لون أو أيقونة تميزه عن الموظفين العاديين

2. **عند فتح ملفه**:
   - التبويبات الأساسية: ملخص | مستندات | إجازات | حضور | رواتب | عهد | سلف
   - **تبويب إضافي: المندوب**
     - معلومات المركبة
     - نوع العقد (راتب/عمولة)
     - نسبة العمولة (إن وجدت)
     - رابط المحفظة

3. **عند الانتقال لأي قسم**:
   - يبقى اسم الموظف في الأعلى
   - كل البيانات تتعلق به فقط
   - زر "عودة للملف" واضح

---

## هـ. كيف سيبدو النظام بعد الإصلاح — للمناديب الخارجيين؟

**من هو المندوب الخارجي؟**
- مندوب يعمل خارج المستودع
- يظهر في **قائمة مناديب العمليات**
- له محفظة كاملة
- يمكن تعيينه في الشحنات

**التجربة الجديدة:**

1. **في قائمة الموظفين**:
   - يظهر بعلامة "مندوب خارجي"
   - أيقونة توضح أنه نشط في العمليات

2. **عند فتح ملفه**:
   - جميع تبويبات المندوب الداخلي
   - **تبويب إضافي: العمليات**
     - عدد الشحنات الحالية
     - رصيد المحفظة
     - رابط مباشر لصفحة المندوب في العمليات

3. **الربط مع العمليات**:
   - عند فتح صفحة المندوب في HR — ترى ملخص عملياته
   - عند فتح صفحة المندوب في العمليات — ترى رابط لملفه في HR

---

## و. كيف سيبدو النظام بعد الإصلاح — لموظفي المكتب؟

**التجربة الجديدة:**

1. **في قائمة الموظفين**:
   - يظهر بعلامة "موظف مكتب"

2. **عند فتح ملفه**:
   - التبويبات الأساسية فقط: ملخص | مستندات | إجازات | حضور | رواتب | عهد | سلف
   - **لا يوجد تبويب مندوب** (ليس مندوباً)
   - بيانات الراتب الثابت فقط

3. **التجربة المبسطة**:
   - لا معلومات إضافية لا يحتاجها
   - واجهة أنظف وأبسط

```sql
CASE p_employee_type
  WHEN 'office' THEN 'office'
  WHEN 'courier_internal' THEN 'courier_commission'  -- ← Maps to commission!
  WHEN 'courier_external' THEN 'courier_3p'
```

### Consequence Analysis

#### Option A: UI is Correct (courier_internal should be salary)
- **Impact**: DB salary calculation uses `courier_commission` logic
- **Result**: Internal couriers get commission-based pay instead of fixed salary
- **Affected**: Payroll RPCs, salary slip generation

- **Impact**: UI displays wrong type label
- **Result**: Confusion in payroll interface, wrong expectations
- **Affected**: Payroll page display only
- **Fix**: Change UI mapping from `courier_salary` to `courier_commission`

#### Option C: Both are Wrong (needs third mapping)
- **Impact**: Requires new staff_role value
- **Result**: Major schema change
- **Fix**: Add new role, migrate data

### Explicit Recommendation

> **Recommend Option A: Fix the Database**
> 
> **Reasoning:**
> 1. Business logic suggests internal couriers (depot workers) are salary-based
> 2. External couriers (field workers) are commission-based
> 3. UI label "مندوب داخلي" implies fixed employment, not per-delivery
> 4. Changing DB is safer than changing UI expectations
>

> **Change:** `WHEN 'courier_internal' THEN 'courier_salary'`

## Prohibition 1: Database Migrations

### Command
**NEVER modify an existing database migration file. Always create a new migration.**

### Why This Prohibition Exists
Database migrations are the immutable history of schema changes. Once applied to any environment, modifying them creates divergence between environments that cannot be reconciled.

### Evidence
- Migration `00090_UNIFIED_STAFF_SYSTEM.sql` defines the unified identity model
- Migration `00059_HR_SYSTEM.sql` defines base HR tables
- Migration `00098_HR_UNIFIED_EMPLOYEE_FINANCIAL_ITEMS.sql` defines financial items

### Damage From Violation
- Production database state becomes irreconcilable with migration history
- New developers cannot recreate the database from migrations
- Rollback operations become impossible
- Data corruption risk during deployments

### Exception Process
If a schema change is required:
1. Create new migration with `supabase migration new`
2. Use `ALTER` statements to modify existing objects
3. Never edit files in `supabase/migrations/` that have been committed
